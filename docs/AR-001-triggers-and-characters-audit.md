## 1. Как работать с этим документом

Это **НЕ Change Request**. Здесь нет готового решения, которое нужно имплементировать. Здесь есть **гипотезы о проблемах**, возникшие при ручном анализе одной партии, и задача на их проверку по коду и дополнительным данным.

**Что требуется от агента:**

1. Прочитать каждое замечание и гипотезу ниже.
2. **По базе кода проверить**, подтверждается ли гипотеза или нет. Если не подтверждается — чётко объяснить почему (ссылка на код, контрпример).
3. Для подтверждённых проблем — оценить **серьёзность** (блокер / важно / косметика) и **сложность фикса**.
4. Предложить, **какие из проблем стоит оформлять как CR-002, CR-003 и т.д.**, а какие можно закрыть точечными фиксами без формального CR.
5. Выдать **итоговый отчёт** по шаблону из раздела 5 этого документа.

**Чего делать НЕ надо:**

- Писать код для фиксов. Это аналитика, не реализация.
- Править PRD или CR-001. Они остаются источниками правды.
- Предлагать новые фичи, не связанные с перечисленными замечаниями.
- Слепо соглашаться с гипотезами. Если по коду видно, что замечание необоснованно — так и писать.

**Важно о принципе работы:** замечания в этом документе получены из **анализа одной партии**. Это маленькая выборка. Агент должен учитывать, что часть симптомов может быть случайностью этой конкретной партии, а не системной проблемой. Для проверок, где одной партии недостаточно — агент должен явно рекомендовать прогнать серию партий и собрать статистику.

## 3. Замечания и гипотезы для анализа

Каждое замечание имеет формат:
- **Наблюдение** — что видно в логе
- **Гипотеза** — предполагаемая причина
- **Вопросы для проверки по коду** — что конкретно искать в имплементации

### 3.1 Триггеры выродились до одного типа

**Наблюдение.**
В `triggered_events` три записи, все три:
```
condition_type: silent_for_n_turns
reaction_type: point_out_inconsistency
```
Хотя в PRD v0.3 раздел 5.3 описано 5 типов условий и 7 типов реакций, и у каждого из 8 персонажей свои триггеры в профиле. За всю партию **ни разу не сработали** `direct_accusation`, `dodged_direct_question`, `repeated_accusation_on_same_target`, `contradiction_with_previous_answer`.

Притом в партии были события, которые должны были активировать эти триггеры:
- Ход 7: Марго прямо обвиняет Штейна ("слова уклончивы, мало конкретики"). Если у Штейна в профиле есть реакция на `direct_accusation`, она должна была рассматриваться системой; если нет — этот пример из списка снимается.
- Ход 10: Зоя прямо обвиняет Марго ("что-то скрываешь"). Должен был сработать `direct_accusation` у Марго (по профилю Марго этот триггер есть, priority 8, threshold 0.4 → `deflect_suspicion_to_another`).
- Ходы 18 + 20: двое подряд голосуют против Штейна. Должен был сработать `repeated_accusation_on_same_target`.

**Гипотеза.** Возможно одно из трёх:
- (a) Часть `condition_type` реализована формально, но условия детекции слишком строгие (например, `direct_accusation` требует упоминания имени + маркеров обвинения, а Зоя говорит "ты что-то скрываешь" без имени);
- (b) `silent_for_n_turns` имеет слишком низкий порог / слишком высокий приоритет и перекрывает все остальные триггеры, которые тоже срабатывают, но проигрывают конкуренцию;
- (c) Часть `condition_type` из enum вообще не имеет имплементации детектора — enum заведён, а код проверки не написан.

**Вопросы для проверки по коду:**
- Для каждого из 5 `condition_type` — есть ли функция-детектор в коде? Где она вызывается?
- На каждый ход `main_round` после ответа — проверяются ли **все** условия, или только те, которые есть у активных персонажей?
- При одновременном срабатывании нескольких триггеров — как выбирается победитель? По `priority` из профиля или глобально?
- Что пишется в `triggered_events` — все сработавшие триггеры или только выигравший?

**Требуемые дополнительные данные:**
- Прогон 10–15 партий на текущем коде, статистика по `triggered_events`: какие `condition_type` встречаются и с какой частотой.
- Если окажется, что в 90%+ случаев это `silent_for_n_turns` — гипотеза (b) подтверждается.
- Если какие-то `condition_type` не встречаются ни разу — гипотеза (a) или (c).

### 3.2 Reaction_type одинаковый у всех сработавших триггеров

**Наблюдение.**
Все три срабатывания имеют `reaction_type: point_out_inconsistency`. При этом по профилям:
- У Марго на `silent_for_n_turns` триггера НЕТ вообще. Её реакции в профиле: `deflect_suspicion_to_another` (на `direct_accusation` и `repeated_accusation_on_same_target`).
- У Зои на `silent_for_n_turns` триггера НЕТ вообще. Её реакции: `mock_with_dry_sarcasm`.
- У Штейна на `silent_for_n_turns` есть (priority 6, threshold 0.7), **и его реакция — `point_out_inconsistency`**. Это совпадает с логом.

По текущему чтению профилей выглядит так, будто Марго и Зоя сработали на триггер, которого у них в профиле нет, с реакцией, которой у них в профиле тоже нет. Однако если в `trigger_rules.json` или в коде есть глобальная надстройка — интерпретация может быть другой, и её нужно проверить отдельно.

**Гипотеза.** Возможно одно из трёх:
- (a) Триггер `silent_for_n_turns` реализован как **глобальный** (в `trigger_rules.json`), а не персональный, и его `reaction_type` выбирается глобально, игнорируя профили персонажей. То есть "молчал N ходов" срабатывает на кого угодно, не только на тех, у кого он в профиле.
- (b) При срабатывании глобального триггера берётся дефолтная реакция, а не реакция из профиля конкретного персонажа.
- (c) Всё сделано верно по v0.3, а я неправильно понял PRD и `silent_for_n_turns` действительно глобальный.

**Вопросы для проверки по коду:**
- Где описаны триггеры для проверки: только в `characters/*.json` или ещё и в `trigger_rules.json`? Что содержит последний?
- При срабатывании `silent_for_n_turns` — как выбирается персонаж, который "влезает"? Из тех, у кого этот триггер в профиле, или любой молчавший N ходов?
- При срабатывании — откуда берётся `reaction_type` для записи в лог и для формирования промпта вмешательства?
- Соответствует ли реальная логика тому, что описано в PRD разделе 5.3 (персональные триггеры — часть профиля) и 8.1-8.8 (у каждого персонажа свой список)?

**Требуемые дополнительные данные:**
- Не нужны, хватает анализа кода и одной партии.

### 3.3 `silent_for_n_turns` как dominant strategy в оркестрации

**Наблюдение.**
Все три срабатывания в этой партии — `silent_for_n_turns`. Паттерн срабатываний по ходам: ход 6 (Марго — до этого молчала), ход 9 (Зоя — до этого отвечала и спрашивала, но потом замолчала), ход 12 (Штейн — то же самое).

То есть **триггер срабатывает в том порядке, в котором персонажи молчат**, и другие триггеры не получают шанс сработать, потому что:
- либо `silent_for_n_turns` проверяется раньше остальных и short-circuit логика выходит;
- либо у него приоритет выше;
- либо остальные триггеры не срабатывают по другой причине (см. 3.1).

**Гипотеза.** `silent_for_n_turns` — это **fallback-триггер**, который должен срабатывать только когда других триггеров нет. Он нужен, чтобы молчащий персонаж не пропадал из партии, а не чтобы перекрывать все остальные условия. Сейчас (гипотетически) он работает наоборот.

**Вопросы для проверки по коду:**
- В каком порядке проверяются триггеры? По приоритету, по индексу в списке, по чему-то ещё?
- Есть ли какой-то механизм "fallback если ничего не сработало"? Или все триггеры равноправны?
- Если у `silent_for_n_turns` глобально задан приоритет выше других — это сознательное решение или дефолт?

**Требуемые дополнительные данные:**
- Серия партий для подтверждения паттерна: если в 10 партиях подряд 80%+ срабатываний — `silent_for_n_turns`, гипотеза подтверждена.

**Связь с другими замечаниями.** Если 3.1 и 3.2 — это про "другие триггеры не реализованы", то это замечание не отдельное, а следствие. Но если 3.1/3.2 разрешатся как "всё реализовано, но проигрывает конкуренцию" — это замечание становится основным.

### 3.4 Автоматическая проверка MUST-директив не отражена в логе

**Наблюдение.**
По PRD раздел 14.2, каждая партия должна содержать автоматическую проверку: "каждый персонаж минимум 2 раза проявил свой MUST-паттерн", "не было 2+ подряд немаркированных реплик". В логе партии **никаких следов этой проверки нет** — ни в `triggered_events`, ни в отдельной секции, ни в `compressed_history`.

При этом по ручной проверке MUST фактически соблюдены:
- Зоя должна "минимум дважды высмеять" — ходы 2 и 15, MUST выполнен.
- Штейн должен "в каждом ответе дать минимум одно сравнение или аналогию" — ходы 6, 20, 26, выполнен.
- Марго должна "не обвинять напрямую, указывать на странное" — ход 7, выполнен.
- Аврора должна "минимум раз за партию превратить ответ в монолог" — ходы 4, 12, выполнен.

**Гипотеза.**
- (a) В модели `Character` нет поля `detectable_markers` (из PRD раздел 5.1 и 8.0), MUST-директивы записаны только как текст в промпте, автоматической проверки не существует.
- (b) Проверка реализована, но её результаты не попадают в лог партии.
- (c) Проверка реализована, но запускается отдельно (например, только по cli-команде post-factum), не автоматически.

**Вопросы для проверки по коду:**
- Есть ли в `Character` поле `detectable_markers`? Если да — заполнено ли оно для всех 8 персонажей?
- Есть ли тип `Marker` (из PRD раздел 5.3.1) и методы `regex` / `counter` / `binary_llm`?
- Выполняется ли проверка MUST в конце партии или между ходами? Если да — куда пишутся результаты?

**Требуемые дополнительные данные:**
- Не нужны, достаточно осмотра кода.

**Серьёзность.** Это **не блокер для MVP** — партия уже играется, характеры проявляются. Но это **блокер для фазы 3 "калибровка"** из PRD раздел 9, потому что без автоматической проверки MUST тюнинг промптов и порогов можно делать только на глаз, что не масштабируется.

### 3.5 Гомогенный каст партии ослабляет контраст

**Наблюдение.**
В партии: Марго (манипулятор, витиеватая), Аврора (драма-квин, витиеватая), Штейн (интеллектуал, витиеватый), Зоя (циник, короткая). Три из четырёх — "витиеватые говоруны". В результате партия местами читается как один длинный поток обтекаемой прозы, в котором Зоя — единственный контраст.

Борис, Ким, Отец Игнатий, Лёха, которые бы дали резкие, нервные, морализаторские, простые реплики — не попали в партию.

**Гипотеза.** Отбор 4 из 8 делается случайно, без балансировки по "энергетическим классам". При малой выборке это даёт перекос.

**Вопросы для проверки по коду:**
- Как именно отбираются персонажи в партию? Полная случайность или есть логика балансировки?
- Есть ли у персонажей какая-то классификация (архетип, темперамент), по которой можно балансировать?

**Это замечание — не баг, а предложение.** Текущая случайная выборка **работает корректно по PRD**. Вопрос в том, хочет ли пользователь добавить логику вида "при отборе N персонажей брать из разных архетипических групп, если возможно". Это потенциальный CR, но только если подтвердится на серии партий, что гомогенные составы — частое явление.

**Требуемые дополнительные данные:**
- Серия 20+ партий, анализ составов: часто ли получаются перекошенные касты (3+ персонажей одного архетипа).

### 3.6 Финальное голосование (F13) не было проверено этой партией

**Наблюдение.**
В логе: `final_vote_result: null`, `vote_changes: []`. Партия завершилась через `spy_guess` раньше, чем дошла до финального переголосования.

Это означает, что CR-001 в части F13 (финальное голосование с возможностью изменения голоса) **не протестирован в боевых условиях**. В логе видно только, что F11 (предварительное) и F12 (защиты) отработали.

**Гипотеза.** Никакой гипотезы о баге нет. Просто фича нетестирована на реальных данных.

**Вопросы для проверки по коду:**
- Есть ли юнит/интеграционные тесты на F13?
- В коде реализованы ли все сценарии из F13: (а) защита состоялась → переголосование; (б) защита пропущена → копирование; (в) все воздержались → шпион выигрывает?

**Требуемые дополнительные данные:**
- Хотя бы одна партия, в которой шпион НЕ угадывает локацию и партия доходит до `final_vote`. Можно искусственно снизить уверенность шпиона через env на период теста.

**Серьёзность.** Важно, потому что F13 — большая часть CR-001, и без её проверки нельзя утверждать, что CR реализован полностью.

### 3.7 Дополнительные наблюдения (низкий приоритет)

Эти точки не требуют глубокого анализа, но стоит упомянуть на случай, если при разборе основных замечаний обнаружатся связанные проблемы:

**(a)** В `spy_confidence_log` отсутствуют записи между ходом 6 и ходом 28 — интервал 22 хода без проверки. По PRD раздел 10.1, `SPY_CONFIDENCE_CHECK_EVERY_N = 3` по дефолту, и на таком интервале ожидалось заметно больше одной проверки. Точное число зависит от того, на каких именно типах ходов запускается проверка (только `answer`? или любые ходы?) — это и предмет уточнения.

**(b)** В `compressed_history` есть сжатая история. По PRD раздел 11.5, это механизм против раздутия контекста. Проверить: срабатывает ли компрессия по порогу, как она влияет на контекст вмешательств и голосований?

**(c)** `display_delay_ms` в диапазоне 1272–11931 мс. Средняя реплика занимает 5–7 секунд. По PRD `SPEECH_DELAY_MULTIPLIER = 0.03` сек/символ — кажется, умножение идёт корректно, но формула могла расходиться с задуманной. Проверить, если будет время.

## 4. Приоритизация для анализа

5. **3.6** (F13 нетестирован) — требует прогонки новой партии с изменёнными параметрами.
6. **3.5** (гомогенный каст) — низкий приоритет, вероятно не проблема, а предложение.
7. **3.7** (дополнительные) — только если останется время после основных пунктов.
