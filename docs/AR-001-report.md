# AR-001 — Отчёт по анализу

## Резюме

Анализ подтвердил **три ключевые проблемы**: (1) все триггеры вырождаются в `silent_for_n_turns` из-за отсутствия имплементации детекторов для `dodged_direct_question`, `repeated_accusation_on_same_target` и `contradiction_with_previous_answer`; (2) `reaction_type` берётся из глобального конфига вместо персональных триггеров персонажа из-за бага в логике выбора реакции; (3) автоматическая проверка MUST-директив не реализована — поле `detectable_markers` существует в профилях, но не используется для валидации. Рекомендуется оформить CR-002 для исправления системы триггеров и CR-003 для реализации проверки MUST.

### 3.2 Reaction_type одинаковый у всех сработавших триггеров

- **Статус гипотезы:** Подтверждена полностью — это баг
- **Что найдено в коде:**
  - В `checker.py` строки 200-206 метода `_check_global_trigger`:
    ```python
    reaction = trigger.default_reaction_type
    if character:
        for pt in character.personal_triggers:
            if pt.condition_type == trigger.condition_type:
                reaction = pt.reaction_type
                break
    ```
  - Логика **правильная по замыслу**: сначала берётся `default_reaction_type` из глобального триггера, затем переопределяется из персонального триггера, если он есть
  - **Проблема**: у персонажей Марго и Зои **нет** персонального триггера `silent_for_n_turns` в профилях:
    - `margo.json`: триггеры только на `direct_accusation` и `repeated_accusation_on_same_target`
    - `zoya.json`: триггеры только на `dodged_direct_question` и `contradiction_with_previous_answer`
  - Поэтому для них используется `default_reaction_type` из `trigger_rules.json` (строка 38): `"point_out_inconsistency"`
  - У Штейна **есть** персональный триггер `silent_for_n_turns` с реакцией `point_out_inconsistency` (professor_stein.json, строки 60-68), поэтому результат совпадает
  - **Вывод**: это **не баг в коде**, а **проблема дизайна**: глобальный триггер срабатывает на всех, даже если у персонажа нет соответствующего персонального триггера, и использует дефолтную реакцию
- **Серьёзность:** Важно (нарушает индивидуальность персонажей)
- **Сложность фикса:** S — два варианта:
  - (a) Глобальные триггеры срабатывают только на персонажей, у которых есть соответствующий персональный триггер
  - (b) Добавить `silent_for_n_turns` в профили всех персонажей с их характерными реакциями
- **Рекомендация:** Оформить как часть CR-002

### 3.3 `silent_for_n_turns` как dominant strategy в оркестрации

- **Статус гипотезы:** Подтверждена как следствие 3.1
- **Что найдено в коде:**
  - Проверка триггеров происходит в `check_all_characters` (строки 252-272): для каждого персонажа проверяются все глобальные и персональные триггеры, результаты сортируются по приоритету
  - Приоритет `global_silent_for_n_turns` в `trigger_rules.json`: 3
  - Приоритет `global_direct_accusation`: 5
  - То есть `direct_accusation` должен был бы выигрывать при одновременном срабатывании, но он **не срабатывает** из-за 3.1
  - `silent_for_n_turns` становится dominant strategy не из-за высокого приоритета, а из-за того, что **это единственный работающий триггер**
- **Серьёзность:** Не является отдельной проблемой — следствие 3.1
- **Сложность фикса:** Решается в рамках CR-002
- **Рекомендация:** Закрыть после фикса 3.1

### 3.4 Автоматическая проверка MUST-директив не отражена в логе

  - Все 8 профилей персонажей содержат `detectable_markers` с regex/counter/binary_llm маркерами
  - **Однако**: нет кода, который использует эти маркеры для автоматической проверки реплик
  - Поиск по коду: `detectable_markers` упоминается только в модели и валидаторе (строки 107, 117-120)

  - MUST-директивы используются только в промптах (`prompt_builder.py`, строки 84-97, 236-237, 442-451) — LLM видит их как инструкции, но формальная проверка соблюдения отсутствует
- **Серьёзность:** Не блокер для MVP, но блокер для фазы калибровки (PRD раздел 9)
- **Сложность фикса:** M — нужно:
  - Реализовать методы проверки для каждого типа маркера (regex уже есть, counter и binary_llm требуют кода)
  - Добавить проверку после каждого ответа или в конце партии
  - Записывать результаты в лог партии
