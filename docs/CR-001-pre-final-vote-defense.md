## 2. Контекст и мотивация

### 2.1 Проблема

В текущей реализации (PRD v0.3) финальное голосование устроено как единственный узловой момент: игроки голосуют → результат определяется → партия заканчивается. У обвиняемого нет окна, в котором он мог бы переломить ситуацию. Это обедняет драматургию: самый напряжённый момент партии (голосование) проходит без взаимодействия.

### 2.2 Что добавляем

Между предварительным и финальным голосованием — **фаза защитных реплик**. Все игроки, получившие максимальное число голосов в предварительном голосовании (при условии, что это число ≥ 2), получают одну короткую реплику на защиту. После всех защит — финальное переголосование.

### 2.3 Почему именно так

- Это **проявляет характеры под максимальным давлением** — именно тот момент, ради которого продукт существует как шоу.
- Это **даёт возможность шпиону выкрутиться риторикой**, а не только угадыванием локации — расширяет пространство игры.
- Это **даёт возможность мирным игрокам передумать** после убедительного/неубедительного выступления — добавляет непредсказуемости финала.
- Это **не ломает жанр Spyfall**: защитная речь — естественное продолжение логики игры, а не чужая механика.

## 3. In scope

Добавить в партию новую фазу `pre_final_vote_defense`, вставляемую между `preliminary_vote` (новая) и `final_vote` (уже существует).

Новый поток фаз партии:

```
setup → main_round → preliminary_vote → pre_final_vote_defense → final_vote → resolution
```

Подсистемы, которые добавляются или меняются:

- Новая фаза **предварительного голосования** (до этого CR финальное голосование было единственным).
- Новая фаза **защитных реплик** для игроков, набравших максимум голосов в предварительном (при минимуме 2).
- Финальное голосование теперь выполняется **после** защит, с возможностью для каждого голосующего изменить свой голос.

---

## 4. Out of scope (жёсткие границы)

Следующее **НЕ делаем** в рамках этого CR:

- **Фаза переговоров / договоров до голосования** — она дублирует основной раунд вопросов и ведёт к раздуванию продолжительности партии. Обсуждение должно происходить в `main_round`.
- **Многоступенчатые защиты / реплики на реплики** — защита одна на каждого подходящего игрока, длительностью не более одной реплики. Никаких "ответов на защиту", никаких "вопросов из зала".
- **Торги / сделки / альянсы** — AI-агенты не ведут себя как политики, дополнительные механики на эту тему не работают (см. решение из базового PRD — "не давать агенту опций, которыми он не воспользуется").
- **Защита для шпиона с правом угадать локацию** — если шпион попал в защитную фазу, он играет её как обычный защищающийся игрок. Право угадать локацию остаётся отдельной механикой из F5 базового PRD и не активируется внутри защиты.
- **Изменение механики досрочного голосования** (F6 базового PRD) — правила запуска досрочного голосования сохраняются, но при запуске оно теперь тоже проходит через новый поток (preliminary → defense → final).

## 5. Изменения в потоке фаз

### 5.1 Было (PRD v0.3, F6)

```
setup → main_round → [optional early vote trigger] → final_vote → resolution
```

Финальное голосование было единым актом: агенты голосуют один раз, результат определяется, партия заканчивается.

### 5.2 Стало

```
setup → main_round → [optional early vote trigger]
      → preliminary_vote
      → pre_final_vote_defense  (только если есть игроки с максимумом голосов ≥ 2)
      → final_vote
      → resolution
```

**Важные инварианты:**

- Если в предварительном голосовании **максимальное число голосов, полученных одним игроком, ниже порога** `DEFENSE_MIN_VOTES_TO_QUALIFY` (дефолт 2) — фаза защит пропускается, финальное голосование проходит сразу по тому же списку голосов (без переголосования).
- Если порог достигнут — защиту получают **все игроки с максимальным числом голосов**. Например, при раскладе 2-2-2 против трёх разных — защищаются все трое.
- Порядок защит между несколькими защищающимися — **случайный** (фиксируется в логе).
- Каждый защищающийся говорит ровно один раз.
- После всех защит **запускается финальное голосование**, где каждый голосующий агент может подтвердить или изменить свой голос.
- Досрочное голосование (из F6 базового PRD) теперь тоже проходит через эту последовательность — `early_vote_trigger → preliminary_vote → (defense) → final_vote`.

## 6. Новая функциональность (acceptance criteria)

### F11. Предварительное голосование

Новая фаза, выполняемая перед защитой и финальным голосованием.

**Acceptance criteria:**

- Каждый игрок в партии (включая шпиона) обязан проголосовать против одного другого игрока или явно воздержаться. **Поведение зависит от env `DEFENSE_ALLOW_ABSTAIN`:**
  - если `true` (дефолт): воздержание разрешено, `target_id == null` — допустимое значение голоса;
  - если `false`: воздержание запрещено, агент обязан выбрать цель из списка кандидатов. Если утилитарная модель всё же вернёт `null` — результат перезапрашивается один раз с жёстким напоминанием в промпте ("ты обязан выбрать одного игрока, воздержание недоступно"); если и повторный запрос вернул `null` — оркестратор выбирает цель случайным образом из доступных кандидатов и пишет в лог предупреждение.
- Голос формируется отдельным вызовом утилитарной модели для каждого агента с контекстом, достаточным для обоснованного выбора (минимум — последние N ходов, где N задаётся конфигом; возможны расширения вида summary + последние K ходов).
- В промпте голосования упоминание опции воздержания должно быть согласовано со значением `DEFENSE_ALLOW_ABSTAIN`: если `false` — опция воздержания в промпт не включается.
- Результат фазы: словарь `{voter_id → target_id | null}` и агрегат `{target_id → count}`.
- **Правило подсчёта:** воздержавшиеся (`target_id == null`) не учитываются в агрегате `{target_id → count}`. То есть если из 6 игроков 3 проголосовали против Бориса и 3 воздержались — у Бориса 3 голоса, максимум по партии — 3.
- Фаза завершается сразу после сбора всех голосов.

### F12. Фаза защитных реплик

Новая фаза, активируемая условно.

**Acceptance criteria:**

- Активируется, если в предварительном голосовании есть хотя бы один игрок, против которого проголосовало **не менее `DEFENSE_MIN_VOTES_TO_QUALIFY` голосов** (дефолт — 2).
- Защиту получают **все игроки, у которых число голосов равно максимальному** в предварительном голосовании (при условии, что этот максимум ≥ порога). То есть:
  - 3-2-1 → защищается один (с 3 голосами)
  - 2-2-2 → защищаются трое (у всех максимум 2)
  - 3-3-1 → защищаются двое
  - 2-2-1-1 → защищаются двое (с 2 голосами)
  - 1-1-1-1-1-1 → максимум 1, ниже порога, фаза защиты пропускается
- Если фаза пропускается (максимум голосов ниже порога) — партия переходит сразу в `final_vote`, который формально копирует результат предварительного.
- Порядок защит: при нескольких защищающихся — случайный порядок (фиксируется в логе для воспроизводимости). Порядок не основан на количестве голосов, потому что при равенстве оно одинаковое.
- Каждый защищающийся получает **одну короткую защитную реплику** длиной **не более `DEFENSE_SPEECH_MAX_SENTENCES` предложений** (ограничение жёсткое, проверяется пост-чеком; если агент выдал больше — обрезается до первых `DEFENSE_SPEECH_MAX_SENTENCES` предложений и в лог пишется предупреждение).
- Реплика формируется через основной (не утилитарный) LLM, промпт содержит: профиль персонажа + контекст партии + список тех, кто против него голосовал + явная установка "тебя обвиняют, у тебя одна короткая защитная реплика — максимум `DEFENSE_SPEECH_MAX_SENTENCES` предложений".
- Реплика **не адресована конкретному игроку** — она направлена всем (`addressee_id = "all"`).
- К защитной реплике применяется та же эмуляция задержки проговаривания (F7 базового PRD), что и к обычным репликам.
- После защитной реплики **не запускается проверка триггеров и окна вмешательства** (F4 базового PRD). Защита — атомарный акт, никто не может её прервать.
- Если защищающихся несколько — между защитами также **не запускаются триггеры**. Все защиты идут подряд, после чего — переход в `final_vote`.

### F13. Финальное голосование с возможностью изменения голоса

Модификация существующей фичи F6 базового PRD.

**Acceptance criteria:**

- **Если фаза защиты была пропущена** (максимум голосов в предварительном ниже `DEFENSE_MIN_VOTES_TO_QUALIFY`): финальное голосование **не выполняется как отдельная фаза с вызовами LLM**. Формально:
  - `final_vote_result = preliminary_vote_result` (прямое копирование)
  - `vote_changes = []` (никто голос не менял)
  - никаких вызовов утилитарной модели в этом случае не производится
  - в `phase_transitions` у записи фазы `final_vote` проставляется `status: "skipped_copied_from_preliminary"`. Это **значение поля status у фазы**, не новое значение `Turn.type` — никаких новых типов ходов в лог не добавляется, список `Turn.type` ограничен значениями из раздела 7.2.
- **Если фаза защиты состоялась**: каждый голосующий агент получает новый вызов утилитарной модели с контекстом: исходный голос + содержимое защитных реплик + вопрос "подтверждаешь ли ты свой голос или меняешь, и если меняешь — на кого".
- Правила относительно воздержания в финальном голосовании **такие же, как в F11**: зависят от `DEFENSE_ALLOW_ABSTAIN`, с той же логикой перезапроса и fallback на случайную цель.
- Воздержавшиеся в предварительном голосовании имеют право проголосовать в финальном (защита могла их убедить) — это актуально только при `DEFENSE_ALLOW_ABSTAIN=true`, так как при `false` воздержавшихся нет по определению.
- **Правило подсчёта финального результата (такое же, как в F11):** воздержавшиеся не учитываются в агрегате `{target_id → count}`.
- **Определение победителя:**
  - если **все финальные голоса — воздержания** (агрегат пуст): шпион выигрывает автоматически;
  - если один игрок получил строго больше голосов, чем любой другой — он считается "выбранным" толпой:
    - если это шпион → мирные выиграли;
    - если это не шпион → шпион выиграл;
  - если за максимум голосов претендуют двое и более: шпион выигрывает автоматически (мирные не смогли определиться, логика Spyfall).

## 7. Точечные изменения в существующих разделах PRD

### 7.1 Раздел 3.6 (F6. Фазы партии и досрочное голосование)

В перечень фаз добавить `preliminary_vote` и `pre_final_vote_defense` между `main_round` и `final_vote`. Обновить описание триггера досрочного голосования — он теперь переводит партию в `preliminary_vote`, а не сразу в `final_vote`.

### 7.2 Раздел 5.4 (модель данных Game → Turn)

В enum `Turn.type` добавить новые значения:

- `preliminary_vote` — запись голоса одного игрока в предварительном голосовании
- `defense_speech` — защитная реплика
- `final_vote` — запись голоса в финальном голосовании (существующий тип `vote` при желании можно депрекейтнуть или оставить как алиас для обратной совместимости)

### 7.3 Раздел 5.4 (модель данных Game)

Добавить новые поля в объект Game:

| Поле | Тип | Описание |
|---|---|---|
| `preliminary_vote_result` | `{voter_id: target_id \| null}` | Снимок предварительного голосования |
| `defense_speeches` | `DefenseSpeech[]` | Список защитных реплик (упорядоченный) |
| `final_vote_result` | `{voter_id: target_id \| null}` | Снимок финального голосования |
| `vote_changes` | `VoteChange[]` | Список игроков, изменивших голос между предварительным и финальным, с указанием из/в |

Новые вложенные типы:

**DefenseSpeech:**
| Поле | Тип | Описание |
|---|---|---|
| `defender_id` | string | ID защищающегося персонажа |
| `votes_received` | int | Сколько голосов получил в предварительном |
| `content` | string | Текст защитной реплики |
| `timestamp` | ISO timestamp | |

**VoteChange:**
| Поле | Тип | Описание |
|---|---|---|
| `voter_id` | string | Кто изменил голос |
| `from_target` | string \| null | На кого голосовал в предварительном (null если воздержался) |
| `to_target` | string \| null | На кого проголосовал в финальном (null если воздержался) |

### 7.4 Раздел 10.1 (env-переменные)

Добавить новые параметры:

| Переменная | Тип | Дефолт | Описание |
|---|---|---|---|
| `DEFENSE_MIN_VOTES_TO_QUALIFY` | int | 2 | Минимальное число голосов, которое должен набрать лидер голосования, чтобы запустилась фаза защит. Защиту получают все игроки с максимальным числом голосов при условии, что это число ≥ порога. |
| `DEFENSE_SPEECH_MAX_SENTENCES` | int | 2 | Максимум предложений в защитной реплике |
| `DEFENSE_ALLOW_ABSTAIN` | bool | true | Разрешено ли воздержание в голосованиях. При `false` агент обязан выбрать цель; опция воздержания убирается из промпта, при возврате `null` — один перезапрос, затем случайный выбор. Подробнее — в F11.

### 7.5 Раздел 6 (UI)

Веб-UI должен визуально различать три фазы голосования и защиту:

- Фаза `preliminary_vote` — все голоса собираются и показываются одним блоком, с визуализацией "X против Y".
- Фаза `pre_final_vote_defense` — защитные реплики выделяются отдельным стилем (например, рамка, центрирование, метка "ЗАЩИТА").
- Фаза `final_vote` — голоса показываются с индикатором "изменил голос" / "подтвердил голос" для каждого голосующего.

### 7.6 Раздел 11 (риски)

Добавить новый подпункт **11.8 Защита скатывается в общие фразы или AI-извинения**:

- **Риск:** защитная реплика — это момент, когда LLM особенно склонен уходить в "извиняюсь, но я не шпион" в нейтральном тоне.
- **Меры:**
  - В промпт защиты явно встраиваются MUST-директивы персонажа с напоминанием "защищайся в своём характере, не в нейтральном LLM-голосе".
  - Отдельный бинарный пост-чек утилитарной моделью: "эта реплика характерна для персонажа X?" — если нет, реплика регенерируется один раз.
  - После фазы 3 калибровки, если проблема массовая — добавить few-shot примеры защитных реплик в промпт для каждого персонажа.

### 7.7 Раздел 14 (критерии готовности MVP)

Добавить новые критерии:

- [ ] Фаза `pre_final_vote_defense` корректно активируется, когда максимум голосов в предварительном голосовании ≥ `DEFENSE_MIN_VOTES_TO_QUALIFY`, и пропускается, когда максимум ниже порога.
- [ ] В партиях с равенством голосов (например, 2-2-2) защиту получают **все** игроки с максимальным числом голосов, не только один.
- [ ] Защитная реплика не превышает `DEFENSE_SPEECH_MAX_SENTENCES`, при превышении — корректно обрезается.
- [ ] Во всех партиях, где была фаза защиты, лог содержит `defense_speeches` и `vote_changes` заполненными корректно.

**Диагностическая метрика (не hard acceptance):**

- Доля агентов, меняющих голос после защиты, отслеживается в логах и анализируется на серии партий. Эта метрика **НЕ является условием готовности MVP**, потому что низкий процент смены голоса может означать одно из трёх:
  - защитная реплика слабая и не влияет на решение (проблема, нужен тюнинг);
  - предварительное голосование уже было устойчивым, защите нечего менять (нормальная ситуация);
  - партия хорошая, но агенты по характеру держат первое решение (тоже нормально).
  
  Поэтому цифра используется как **сигнал к ручному анализу**, а не как автоматический pass/fail. Если на серии партий процент смены голоса оказывается устойчиво около нуля — это повод посмотреть логи и понять причину, но не повод считать MVP проваленным.

## 9. Отклонённые альтернативы (зачем они здесь)

Этот раздел фиксирует идеи, которые рассматривались и **сознательно отвергнуты**. Если в будущем возникнет соблазн их вернуть — сначала перечитать, что уже обдумано.

### 9.1 Фаза переговоров/договорённостей ДО голосования — отклонено

**Почему обсуждалось:** дать игрокам возможность "свериться" перед тем, как голосовать.

**Почему отклонено:** дублирует основной раунд вопросов. То, что стоило бы сказать "в переговорах" — должно было прозвучать в `main_round`. Введение отдельной переговорной фазы растягивает партию без добавления новой смысловой работы.

### 9.2 Отбор защищающихся: рассмотрены три варианта, принят третий

**Вариант (a): защиту получает только лидер с наибольшим числом голосов** — отвергнут, потому что не покрывает случаи ничьей (2-2-2 → никто не защищается, хотя под серьёзным подозрением сразу трое).

**Вариант (b): абсолютный порог 50%+ от числа проголосовавших** — отвергнут, потому что ломается на распылённых голосах: при 2-2-2 никто не дотягивает до 50%, фаза пропускается, хотя логически трое под подозрением.

**Вариант (c, принят): защиту получают все игроки с максимальным числом голосов, при минимуме 2 голосов** — корректно обрабатывает и лидерский сценарий (3-2-1 → один защитник), и ничейный (2-2-2 → трое защитников), и распылённый (1-1-1-1-1-1 → фаза пропускается, потому что подозрение настолько размыто, что защищать некого конкретно).

### 9.3 "Все, кто получил хотя бы один голос, получают защиту" — отвергнуто

**Почему обсуждалось:** никто из обвинённых не остаётся без возможности высказаться.

**Почему отклонено:** при 6 игроках и распылённом голосовании может защищаться до 6 человек подряд. Партия растягивается, финал теряет фокус, возвращение к "болоту". Минимум 2 голоса работает как естественный фильтр: "под подозрением" значит, что хотя бы два других игрока независимо выбрали этого.

### 9.4 Многоступенчатые защиты / реплики на реплики — отклонено

**Почему обсуждалось:** ещё больше драмы, возможность ответить на защиту.

**Почему отклонено:** бесконечно удлиняет партию, размывает напряжение финала, вводит механику, которой AI-агенты всё равно не будут пользоваться осмысленно (отсылка к базовому PRD — "не давать агенту опций, которыми он не воспользуется").

### 9.5 Позволить шпиону угадать локацию во время защитной фазы — отклонено

**Почему обсуждалось:** логичный момент для шпиона "раскрыться".

**Почему отклонено:** дублирует механику F5 базового PRD (шкала уверенности шпиона уже даёт ему право угадать в любой момент). Введение этого как отдельной опции в защите расщепляет решение шпиона между двумя механиками и усложняет промпты.
