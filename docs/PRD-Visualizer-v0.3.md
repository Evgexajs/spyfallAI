**Кодовое имя:** Visualizer (рабочее)
**Версия PRD:** 0.3
**Дата:** 2026-04-21
**Статус:** готов к передаче агенту-архитектору для декомпозиции на таски

- Добавлен раздел о структуре проекта

## 1. Контекст и мотивация

Движок игры SpyfallAI генерирует партии AI-персонажей в формате JSON-логов. Логи содержат всё необходимое для воспроизведения партии — реплики по ходам, фазы, голосования, результаты.

До сих пор партии были доступны только как текстовые логи для анализа. Для публичного показа (YouTube, TikTok и т.п.) нужна визуализация: сцена с локацией, персонажи, их реплики в виде облачек с эффектом печатания, смена визуальных состояний по фазам игры.

Цель визуализатора — дать возможность получать **записываемые через screen recording** видеоролики партий для публикации. MVP не решает задачу автоматической генерации видеофайла — первая версия ориентирована на ручную запись экрана (OBS, Screen Studio и т.п.).

## 2. Архитектурная позиция

Визуализатор — **самостоятельная подсистема**, не зависящая от движка игры SpyfallAI.

```
┌─────────────────┐      ┌───────────────┐      ┌───────────────────┐
│  SpyfallAI      │      │ Преобразователь│      │ Визуализатор      │
│  (движок игры)  │──────│ (адаптер)     │──────│ (этот PRD)        │
│  JSON-лог       │      │ → API format  │      │ Читает контракт,  │
│                 │      │               │      │ рендерит сцену    │
└─────────────────┘      └───────────────┘      └───────────────────┘
```

**Ключевые свойства:**

- Визуализатор НЕ знает про SpyfallAI. Он работает с **обобщённым форматом "партия диалогового шоу"**.
- Визуализатор принимает на вход JSON в своём собственном формате (API-контракт из раздела 5), не сырой лог SpyfallAI.
- Преобразователь из формата SpyfallAI в формат визуализатора — **отдельный компонент**, не входит в scope этого PRD.
- Теоретически визуализатор можно использовать для любых диалоговых партий (Мафия, Detective Club и т.п.) — достаточно написать другой адаптер.

Это архитектурное решение зафиксировано и не обсуждается в этом PRD.

## 3. Цели MVP

**Режим работы:**

MVP поддерживает **только offline replay полностью загруженного JSON**. Realtime stream / потоковое воспроизведение событий — **не в scope**.

**В scope:**

- Загрузка партии из JSON-файла (выбор файла через UI)
- Отображение сцены с фоновой картинкой локации
- Отображение персонажей на сцене как визуальных сущностей (в MVP — плейсхолдеры-круги с именем, архитектурно готовые к замене на полноценные изображения)
- Реплики персонажей отображаются как "облачка" над головами
- Эффект "печатания" реплики (текст появляется постепенно, не разом)
- Индикатор "персонаж печатает" (три точки в облачке) до начала реплики
- Воспроизведение партии последовательно по timeline, с разумными паузами между событиями
- Смена визуального состояния сцены при смене фаз партии (main_round → голосование → защита → финал → разрешение)
- Специальная визуализация для события `spy_guess` (ставка шпиона на локацию)
- Финальный экран `outcome`
- Запускается локально в браузере, предназначен для записи через screen recording

**Не в scope:**

- Автоматическая генерация видеофайла (mp4). Только ручная запись экрана.
- TTS-озвучка голосов персонажей
- Фоновая музыка и звуковые эффекты
- Полноценные изображения персонажей (в MVP — плейсхолдеры)
- Динамическая смена ракурсов камеры, "режиссура"
- Интерактивность для зрителей (это не продукт для них, только инструмент съёмки)
- Визуальное выделение роли шпиона до разоблачения
- Адаптер из JSON SpyfallAI в формат визуализатора (отдельная задача)
- Оптимизация под мобильные устройства
- Realtime stream событий

## 4. Технический стек

**Фронтенд:**
- JavaScript (или TypeScript — на усмотрение разработчика)
- PixiJS как основной графический движок для Canvas-рендеринга 2D
- HTML/CSS для обёртки UI (загрузка файла, управление воспроизведением)

**Запуск:**
- Статический сайт, открывается в браузере локально
- Никакого серверного бэкенда в MVP

**Зависимости:**
- PixiJS — рендеринг
- Возможно, небольшая библиотека для анимации tween-ов (опционально, на усмотрение разработчика)

**Обоснование выбора:**
- PixiJS оптимален для 2D Canvas-анимации, меньше boilerplate чем голый Canvas API
- Three.js избыточен для плоской сцены
- HTML/CSS-анимации плохо работают для плавного эффекта печатания и смены сцен
- Отсутствие бэкенда — простота развёртывания и запуска

## 5. Контракт входных данных (API-формат)

Визуализатор принимает один JSON-файл с описанием партии. Формат **не совпадает** с форматом лога SpyfallAI — это обобщённый формат "партия диалогового шоу".

### 5.1 Верхнеуровневая структура

```json
{
  "version": "1.0",
  "metadata": {
    "game_id": "string, идентификатор партии",
    "title": "string, опциональный заголовок партии для интро"
  },
  "scene": {
    "location_id": "string, ID локации (например: 'restaurant', 'submarine')",
    "location_name": "string, читаемое имя локации ('Ресторан')"
  },
  "characters": [
    {
      "id": "string, уникальный ID персонажа",
      "display_name": "string, имя для отображения",
      "position_hint": "integer, индекс слота на сцене (0..N-1)"
    }
  ],
  "timeline": [
    {
      "type": "string, тип события (см. раздел 5.2)",
      "...": "поля зависят от типа"
    }
  ]
}
```

### 5.2 Типы событий в `timeline`

События обрабатываются строго последовательно. Визуализатор не "умный" — он просто рендерит их по порядку.

Поля `type` для всех событий — значения из фиксированного набора.

**Тип `speech` — реплика персонажа:**

```json
{
  "type": "speech",
  "speaker_id": "string, ID персонажа из characters",
  "addressee_id": "string | null, ID адресата или null если ко всем",
  "content": "string, текст реплики",
  "subtype": "normal | defense | post_guess"
}
```

Поле `subtype`:
- `normal` — обычная реплика в ходе партии (вопрос, ответ, реакция)
- `defense` — защитная речь после обвинения
- `post_guess` — реплика после момента угадывания шпионом локации (реакция)

Рендеринг: облачко над персонажем, сначала индикатор печатания (три точки), потом текст печатается символ за символом. Разные `subtype` могут иметь визуальные отличия (например, `defense` с акцентной рамкой облачка) — детали на усмотрение разработчика, но визуальное различие **должно быть заметно**.

**Тип `phase_change` — смена фазы партии:**

```json
{
  "type": "phase_change",
  "phase": "main_round | voting | defense | final | resolution",
  "label": "string, человекочитаемое имя фазы ('Голосование')"
}
```

Рендеринг: меняется визуальное состояние сцены (например, затемнение фона при голосовании, подсветка в финале).

> **Примечание про enum `phase`:** набор значений фиксирован для MVP. Для адаптеров других игр (Мафия, Detective Club и т.п.) enum может расшириться в будущих версиях контракта. Расширение трактуется как **breaking change версии API** и требует инкремента `version` в JSON. Визуализатор при неизвестном значении `phase` возвращает ошибку валидации, а не молчаливый fallback.

**Тип `system_message` — системное сообщение:**

```json
{
  "type": "system_message",
  "content": "string, текст ('Голосование инициировано', 'Осталось 20% времени')"
}
```

Рендеринг: появляется короткий overlay-текст сверху или снизу сцены.

**Тип `vote` — голос персонажа:**

```json
{
  "type": "vote",
  "phase": "preliminary | final",
  "voter_id": "string",
  "target_id": "string, против кого голосует",
  "comment": "string | null, комментарий к голосу или null если тихий"
}
```

Поле `phase`:
- `preliminary` — предварительный голос в ходе голосования
- `final` — окончательный голос (после защитных речей и перераундов)

Рендеринг:
- Если `comment != null` — сначала над `voter_id` появляется speech-like облачко с текстом комментария (работает по тем же правилам печатания и hold-а, что и обычный `speech`)
- После исчезновения облачка (или сразу, если `comment = null`) — визуализация голоса: стрелка от `voter_id` к `target_id`, подсветка цели, иконка
- `final` имеет более выраженный визуальный акцент (например, ярче стрелка, задержка дольше) чем `preliminary`

**Тип `spy_guess` — ставка шпиона на локацию:**

```json
{
  "type": "spy_guess",
  "spy_id": "string, ID того, кто угадывает",
  "guessed_location_id": "string, ID угаданной локации",
  "guessed_location_name": "string, читаемое имя",
  "correct": "boolean, было ли угадывание верным"
}
```

Рендеринг: это **драматичный проходящий момент**, не финальный экран. Шпион визуально выделен (зум, подсветка), на экран выводится его догадка крупно, потом — раскрытие (верно / неверно). После раскрытия — **возврат на сцену** (не финал).

> **Различие `spy_guess` и `outcome`:** `spy_guess` — момент ставки, зрелищный, но проходящий. После него воспроизведение **продолжается** (могут быть реплики `post_guess`, голосования и т.п.). `outcome` — финальный экран, после него воспроизведение заканчивается. Разработчику критично **не путать их и не делать два одинаковых финальных экрана подряд**.

**Тип `outcome` — итог партии:**

```json
{
  "type": "outcome",
  "winner": "spy | civilians",
  "spy_id": "string, ID того, кто был шпионом",
  "reason": "string, человекочитаемое описание исхода"
}
```

Рендеринг: финальный экран с подсветкой шпиона и результатом.

> **`outcome` обязательно — последнее событие в timeline.** Если за `outcome` идут другие события, это ошибка валидации. После `outcome` воспроизведение заканчивается.

### 5.3 Требования к валидации

**Схема:**
- При загрузке JSON — проверка схемы. Если критичные поля отсутствуют или тип неверный — **явная ошибка в UI** с указанием проблемы, а не молчаливый сбой
- Поля, не описанные в контракте, **игнорируются** без ошибки
- `characters` не пустой (минимум 2), максимум 8
- `timeline` не пустой
- Значения enum-полей (`phase`, `subtype`, `winner`, `type`) — строго из допустимого набора, иначе ошибка
- `outcome` обязательно последнее событие в timeline (если присутствует)

**Ссылочная целостность:**

Все ID-ссылки на персонажей в событиях timeline должны указывать на существующие ID из `characters`. Проверяемые поля:

- `speech.speaker_id` — обязательно существует в `characters`
- `speech.addressee_id` — либо `null`, либо существует в `characters`
- `vote.voter_id`, `vote.target_id` — обязательно существуют в `characters`
- `spy_guess.spy_id` — обязательно существует в `characters`
- `outcome.spy_id` — обязательно существует в `characters`

Любая висячая ссылка — ошибка валидации при загрузке, с указанием какое событие и какое поле не прошло проверку.

### 5.4 Принцип последовательной обработки

События timeline проигрываются **строго последовательно**. Следующее событие не начинается, пока текущее не завершилось полностью — включая:
- Hold-время у облачка речи
- Анимацию `phase_change`
- Раскрытие результата в `spy_guess`
- Все переходы и промежуточные паузы

Параллельных событий в MVP нет. Это упрощает логику воспроизведения и делает поведение предсказуемым.

## 6. Функциональные требования MVP

### 6.1 UI-оболочка

- Главная страница с селектором файла (input type=file для JSON)
- Кнопки управления воспроизведением: Play, Pause, Restart
- Индикатор прогресса (какое событие из timeline сейчас проигрывается, сколько осталось)
- Опциональное ускорение/замедление (x0.5, x1, x2) — для удобства записи роликов разной длительности

### 6.2 Тайминги — визуализатор единственный источник

**Жёсткое правило:** в MVP визуализатор полностью самостоятельно вычисляет длительность каждого события по своим внутренним правилам. Входной JSON **не содержит** полей с таймингами, визуализатор их не ожидает и не использует.

Это развязывает адаптер: он просто конвертит события по типу и полям, не думая о длительности воспроизведения.

Внутренние правила таймингов визуализатора (параметры настраиваются конфигом, дефолты ниже):

- **Индикатор печатания** перед речью: длительность = `max(500мс, min(3000мс, 100мс × длина_реплики_в_символах))`
- **Скорость печатания** текста: ~30мс на символ
- **Hold после завершения печати:** 1500мс на каждые 20 символов текста, минимум 1000мс
- **Пауза между событиями** timeline: 500мс по умолчанию
- **Пауза перед голосованием:** +1000мс дополнительно (драматургия)
- **Длительность phase_change анимации:** 1000-2000мс
- **Длительность spy_guess:** 3000-5000мс (большая, центральное событие)
- Умножитель скорости (x0.5 / x1 / x2) **пропорционально** масштабирует все эти значения

### 6.3 Сцена

- Фоновое изображение локации берётся из `assets/locations/{location_id}.png`
- Если картинки нет или не загрузилась — fallback на градиент
- Персонажи расставлены на сцене по `position_hint` — индекс слота из фиксированной карты слотов (см. 6.3.1)
- Масштаб сцены фиксированный (1920x1080 или близко), без адаптивности в MVP

#### 6.3.1 Карты слотов для расстановки персонажей

Визуализатор имеет фиксированные карты слотов для `N = 2..8` персонажей. Каждая карта — массив координат вида `[{x, y}, ...]`, где индекс в массиве соответствует `position_hint`.

Рекомендуемые расстановки (конкретные координаты — на усмотрение разработчика, главное принцип):
- **N=2:** два персонажа по бокам сцены
- **N=3-4:** расстановка треугольником или в ряд
- **N=5-6:** полукруг
- **N=7-8:** два ряда или расширенный полукруг

**Поведение при проблемах с `position_hint`:**
- Если `position_hint` вне диапазона (< 0 или > N-1) — fallback на порядок массива `characters`
- Если два персонажа имеют одинаковый `position_hint` — fallback на порядок массива
- Если `position_hint` отсутствует — fallback на порядок массива
- Любой fallback не является ошибкой загрузки, визуализатор продолжает работу

**Детерминированность:**

При одинаковом входном JSON расстановка персонажей **детерминирована**. Два запуска с одним и тем же файлом дают идентичное размещение. Никакого рандома в позиционировании быть не должно.

### 6.4 Персонажи

- **MVP-плейсхолдер:** круг с именем персонажа по центру. Цвет круга — детерминированный хеш от ID (каждый персонаж имеет свой стабильный цвет)
- Персонаж имеет визуальные состояния: `idle`, `speaking`. В MVP реализовать оба
- Состояние `speaking`: персонаж "активен" (подсветка, лёгкая анимация — например, пульсация масштаба)
- Когда персонаж неактивен — `idle` (нейтральный)
- Поле `addressee_id` в событии `speech` в MVP **не влияет на рендер** (облачко появляется над `speaker_id`, никаких стрелок/подсветки адресата). Поле зарезервировано как задел под будущую фичу подсветки адресата

### 6.5 Облачко с репликой

- Появляется над головой персонажа при `speech` событии (или при `vote` с непустым `comment`)
- Показывается индикатор печатания (анимация из трёх точек) — длительность по правилам 6.2
- Затем текст печатается символ за символом
- После завершения печати облачко остаётся (hold) ещё время по правилам 6.2, потом исчезает
- Одновременно может быть показано только одно облачко (визуализатор обрабатывает события последовательно)
- `speech.subtype` визуально различим (например, `defense` — акцентная рамка облачка, `post_guess` — другой фон). Конкретная реализация на усмотрение разработчика, требование — **визуальное различие должно быть заметно зрителю**
- Облачко `vote.comment` использует нейтральный стиль (ближе к `speech.subtype: normal`), не акцентный

### 6.6 Смена фаз

- При событии `phase_change`:
  - Кратковременная смена визуального состояния (1-2 секунды): затемнение, надпись с именем фазы, подсветка
  - Визуальный фон сцены может меняться в цветовой температуре (голосование — холоднее, финал — контрастно)
- Подробная реализация визуальных эффектов — на усмотрение разработчика, главное — зритель должен **чувствовать, что фаза сменилась**

### 6.7 Угадывание локации шпионом (spy_guess)

- Событие `spy_guess` рендерится как **драматичный проходящий момент**, не как облачко
- Шпион визуально выделяется (зум, подсветка)
- На экран выводится крупно название догадки (`guessed_location_name`)
- После паузы — раскрытие результата: `correct: true` — подсветка совпадения, `correct: false` — визуализация промаха
- После раскрытия — возврат на обычный вид сцены, воспроизведение **продолжается** (это важно — `spy_guess` не финал)
- Это один из самых драматургически важных моментов, его длительность — увеличенная (см. 6.2)

### 6.8 Финальный экран (outcome)

- Событие `outcome` — финальный экран. После его отрисовки воспроизведение **заканчивается**
- Кнопки Play/Pause неактивны, доступен только Restart
- Финальный экран: подсветка `spy_id` (раскрытие роли), текст `reason`, явное указание победителя (`winner`)
- Визуально **должен отличаться от `spy_guess`** — если `spy_guess` это "момент ставки", то `outcome` это "итоговый экран", другая композиция

### 6.9 Поведение Play / Pause / Restart

- **Play:** запускает воспроизведение с текущей позиции timeline. Если timeline не начинался — с начала
- **Pause:** замораживает **текущее состояние ровно как есть** — включая печатание посреди слова, анимацию phase_change на полпути, индикатор печатания. При возобновлении (Play) продолжается с того же места
- **Restart:** полный сброс:
  - Индекс timeline в 0
  - Все облачка удаляются
  - Все transient-эффекты очищаются (phase styling, overlays, активные анимации)
  - Состояния персонажей → `idle`
  - Воспроизведение НЕ начинается автоматически, ждёт нажатия Play

## 7. Архитектурные требования

### 7.1 Разделение слоёв

Визуализатор состоит из чётко разделённых слоёв:

- **Слой парсинга** (`parser`): валидация и разбор JSON в типизированные объекты
- **Слой воспроизведения** (`player`): модель состояния (текущее событие, пауза, скорость, таймеры)
- **Слой рендера** (`render`): PixiJS-сцена, персонажи, облачка, фон, эффекты
- **Слой UI** (`ui`): HTML-обёртка, контролы Play/Pause/Restart, селектор файла, индикатор прогресса

Слои изолированы. Рендер не дёргает парсинг напрямую. Player не знает про PixiJS — только про события и состояние.

### 7.2 Абстракция рендеринга персонажа

Существует интерфейс `CharacterRenderer` с методами типа `render(position, state)`, `setState(state)`, `destroy()`.

Сцена использует **только этот интерфейс**, не зная о конкретной реализации.

В MVP реализация — `PlaceholderCharacterRenderer` (круг с именем). Замена на другую реализацию (например, `SpriteCharacterRenderer` с полноценными картинками) **не требует изменений в сцене, player или парсере** — только в одном месте: фабрике `createCharacterRenderer(characterId)`.

Это архитектурный дизайн-поинт, критичный для будущего развития.

### 7.3 Готовность к добавлению звуков (post-MVP)

Звуковой слой не реализуется в MVP, но в архитектуре должны быть точки интеграции:
- Событие "персонаж начал говорить" — сюда подцепится TTS
- Событие "смена фазы" — сюда подцепится звуковой акцент
- Событие "фоновая атмосфера локации" — сюда подцепится эмбиент
- Событие "spy_guess" — сюда подцепится драматичный звуковой эффект

Просто оставить комментарии и пустые функции в соответствующих местах, чтобы post-MVP не пришлось переписывать архитектуру.

### 7.4 Рекомендуемая структура проекта

Ориентир для архитектора, детали — на его усмотрение:

```
src/
├── parser/      # валидация и разбор JSON
├── player/      # модель воспроизведения, state machine, таймеры
├── render/      # PixiJS слой (сцена, персонажи, облачка, эффекты)
├── ui/          # HTML-обёртка, контролы
└── config/      # константы (тайминги, карты слотов, цвета фаз)
assets/
├── locations/   # картинки локаций (PNG/WebP)
└── fonts/       # шрифты
```

Это **рекомендация**, не жёсткое требование. Архитектор может адаптировать.

### 7.5 Preloading ассетов

- При загрузке JSON визуализатор preload-ит картинку локации из `scene.location_id` и все необходимые шрифты
- До завершения preload-а кнопка Play **disabled**
- В UI отображается индикатор "Loading assets..."
- При ошибке загрузки картинки локации — fallback-градиент, Play **активируется** (не блокировка)
- При ошибке загрузки шрифта — fallback на системный sans-serif

## 8. Ассеты для MVP

**Картинки локаций:** 15 штук, 1920x1080, формат PNG/WebP. Путь: `assets/locations/{location_id}.png`.

- Предполагается генерация через ИИ (Midjourney / Flux / Dall-E) заранее
- Стилистика должна быть консистентной между локациями (одинаковая палитра, одинаковый уровень детализации)
- **Это не задача разработчика визуализатора.** Картинки готовятся отдельно и кладутся в репо. Если на момент MVP не все готовы — работает fallback-градиент

**Картинки персонажей:** в MVP не используются (плейсхолдеры-круги)

**Шрифты:**
- Читаемый sans-serif с поддержкой кириллицы
- Стабильный line-height
- Один основной шрифт для облачек и системных надписей
- Путь: `assets/fonts/`

## 9. Acceptance criteria MVP

1. Визуализатор загружается по локальному URL в браузере
2. Через UI можно загрузить JSON-файл партии
3. JSON проходит валидацию по контракту (раздел 5); некорректный JSON показывает понятную ошибку в UI, включая случаи висячих ссылок на несуществующих персонажей и `outcome` не на последнем месте
4. После загрузки JSON картинка локации preload-ится, до готовности кнопка Play disabled
5. При нажатии Play начинается последовательное проигрывание событий timeline
6. События timeline проигрываются строго последовательно, параллельных событий нет
7. Фон сцены соответствует `scene.location_id` (или fallback-градиент если картинки нет)
8. Все персонажи из `characters` отображаются как плейсхолдеры-круги с именами, на слотах согласно `position_hint` с корректным fallback-поведением
9. Расстановка персонажей детерминирована (одинаковый JSON → одинаковое размещение)
10. При событии `speech` — над говорящим появляется облачко с индикатором печатания, затем текст печатается посимвольно
11. Разные `speech.subtype` (normal / defense / post_guess) визуально различимы
12. При событии `phase_change` — виден визуальный переход между фазами
13. При событии `vote` — если есть `comment`, появляется облачко над voter-ом, после — стрелка/подсветка голоса; разные `phase` (preliminary/final) визуально различимы
14. При событии `spy_guess` — центральный крупный рендер с догадкой и раскрытием; после раскрытия возврат на обычную сцену, воспроизведение **продолжается**
15. При событии `outcome` — финальный экран, визуально отличный от `spy_guess`; воспроизведение заканчивается
16. Pause замораживает текущее состояние ровно в том виде, в каком оно есть (включая печатание посреди слова)
17. Restart полностью сбрасывает состояние (timeline, облачка, эффекты, состояния персонажей)
18. Скорость воспроизведения регулируется (x0.5, x1, x2), пропорционально меняются все тайминги
19. Сцена использует `CharacterRenderer` интерфейс; замена реализации не требует изменений в сцене/player/парсере

## 10. Post-MVP backlog

Для ориентации, что идёт после MVP (не делается сейчас, но архитектура должна быть готова):

**Уровень 2 (ближе к публикации полноценного контента):**
- Полноценные картинки персонажей (2D-спрайты) вместо плейсхолдеров. Минимум 2 состояния на персонажа (idle, speaking)
- Фоновая атмосфера локации (ambient sound). Один лой на локацию, циклический
- Динамическое изменение фоновой сцены при напряжённых фазах (подсветка, тонкие визуальные эффекты)
- Индикатор "кто к кому обращается" (стрелка, подсветка адресата — активация поля `addressee_id`)

**Уровень 3 (продакшн-качество):**
- TTS-озвучка реплик с разными голосами для персонажей
- Динамическая музыка, реагирующая на фазы
- Автоматическая генерация видеофайла (mp4) без screen recording
- Режиссура: динамические ракурсы, зум на говорящего, смена планов
- Визуальные эффекты в `spy_guess` и `outcome` — разоблачение с анимацией

**Развитие API-контракта:**
- Расширение enum `phase` для других игр (Мафия, Detective Club и т.п.) — как breaking change с инкрементом version
- Опциональное поле `position_override` с глобальным slot_id для разных схем сцен (зал суда, бар, круглый стол)

**Неопределённое будущее:**
- Визуальное выделение роли шпиона до разоблачения (механика сейчас не определена, решение после первых роликов)
- Интерактивный режим для зрителей (web-продукт с возможностью смотреть партию в реальном времени)
- Realtime stream событий (вместо offline replay)
- Мобильная адаптация
