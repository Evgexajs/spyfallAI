## 1. Обзор приложения и цели

### 1.1 Суть продукта

**SpyfallAI** — генератор игровых партий в Spyfall, где все игроки являются AI-агентами с фиксированными характерами. Продукт производит наблюдаемый контент: текстовые партии, которые создатель может смотреть в реальном времени и позже использовать для производства роликов / стримов / других форматов потребления.

**Ключевая ценность:** зритель получает не "демо LLM, играющих в игру", а **шоу с характерными персонажами**. Персонажи проявляют ярко выраженные качества (агрессия, манипулятивность, паранойя, морализаторство и т.п.) за счёт **жёстких игровых директив**, а не просто за счёт промпт-описания "ты агрессивный".

### 1.2 Цели MVP

1. Получить стабильное ядро: оркестратор партии, 8 проработанных персонажей, 8–10 локаций с ролями, система триггеров, логирование партий.
2. Генерировать партию длительностью 15–30 минут (настраиваемо) с вменяемой стоимостью (~$1–2 за партию на прод-длительности).
3. Дать создателю минимальный UI для запуска, паузы и остановки партии.
4. Вести полный лог партии для дальнейшего монтажа / анализа / переиспользования.

### 1.3 Что НЕ входит в MVP

- Визуализация персонажей (аватары, анимации, сцена).
- Live-стриминг для сторонних зрителей.
- Авторизация и мультипользовательский доступ.
- Монтаж/склейка роликов внутри системы.
- Агент-нарратор / ведущий.
- Участие живого игрока в партии.
- База данных (используем JSON-файлы).
- Свободное голосование, инициируемое агентами по их воле.

## 2. Целевая аудитория

### 2.1 Основной пользователь MVP

**Создатель контента** (первично — сам автор проекта): один человек, запускающий партии, собирающий логи и готовящий из них ролики / демо / материалы для соцсетей. Технически подкован, умеет работать с CLI и минимальным веб-UI.

### 2.2 Целевой зритель (конечный потребитель контента)

Люди, смотрящие короткие ролики / клипы / стримы с AI-персонажами. Их ожидание — **развлечение через характеры**, не погружение в механику игры. Они должны понимать, что происходит, за 20–30 секунд и залипать на персонажах.

### 2.3 Аудитория за пределами MVP

Потенциально — прямые зрители live-партий, участники-люди, создатели кастомных персонажей / локаций, организаторы турниров AI-агентов.

## 3. Основные функции и функциональность

### 3.1 F1. Оркестратор партии

Запускает и ведёт партию от старта до финала, управляя фазами, ходами, триггерами и завершением.

**Acceptance criteria:**
- Партия запускается командой (CLI или веб-кнопкой) с параметрами: состав персонажей, локация (или случайная), длительность.
- Оркестратор присваивает роль шпиона случайному персонажу, остальным выдаёт локацию и роли.
- Основной цикл: вопрос → ответ → следующий вопрос кому угодно кроме задавшего.
- После каждого **ответа** (не после вопроса) запускается проверка триггеров.
- Партия завершается одним из способов: истёк таймер / достигнут лимит вопросов / запущено досрочное голосование / шпион заявил угадывание локации.
- Оркестратор пишет полный лог (см. F8).

### 3.2 F2. Система персонажей

Фиксированная труппа из 8 персонажей, каждый описан в отдельном JSON-файле. В партию подбирается подмножество (4 на отладке, 6 на проде, максимум 8).

**Acceptance criteria:**
- Каждый персонаж имеет: имя, биографию, набор MUST / MUST NOT директив, список персональных триггеров, стиль речи, приоритет вмешательства.
- Промпт агента собирается из профиля + текущего состояния игры + его секретной информации (шпион он или знает локацию).
- Персонажи не могут выйти из роли (MUST NOT: выходить из образа, признавать, что он AI, нарушать свои директивы).

### 3.3 F3. Система локаций и ролей

Каталог из 8–10 локаций, каждая содержит 3–5 ролей. Хранится в `locations.json`.

**Acceptance criteria:**
- При старте партии случайно (или вручную) выбирается локация.
- Всем игрокам кроме шпиона рассылается: локация + их роль.
- Шпион получает только уведомление "ты шпион, локация неизвестна".

### 3.4 F4. Система триггеров и окон вмешательства

Кодовый слой (не LLM) анализирует последний ответ и проверяет, сработал ли у кого-то из агентов триггер на вмешательство.

**Acceptance criteria:**
- Триггеры делятся на глобальные (одинаковые для всех) и персональные (часть профиля персонажа).
- Триггер задан структурно: `condition_type` (enum), `priority` (int), `threshold` (float), `reaction_type` (enum), опциональные параметры.
- Если у нескольких агентов одновременно сработали триггеры — побеждает тот, у кого выше приоритет (при равенстве — случайный выбор).
- Выигравшему агенту отправляется микро-промпт ("влезаешь? если да — одной фразой"), он может отказаться.
- После вмешательства цикл продолжается с того же места (следующий по очереди вопрос).

### 3.5 F5. Шкала уверенности шпиона

Отдельный внутренний механизм, периодически дающий агенту-шпиону выбор своего состояния без обязательства называть локацию.

**Acceptance criteria:**
- Каждые N ходов (настраиваемо, дефолт — каждые 3 ответа) шпиону отправляется скрытая микро-проверка.
- Агент-шпион выбирает одно из состояний: `no_idea`, `few_guesses`, `confident`.
- Только в состоянии `confident` ему даётся опция на следующем ходу **заявить о угадывании локации**. Это — единственный способ шпиону выиграть досрочно.
- Если шпион заявляет угадывание: называет локацию, партия завершается сразу. Угадал — шпион победил, промазал — проиграл.

### 3.6 F6. Фазы партии и досрочное голосование

Партия проходит через фазы: **setup → main_round → optional_vote → final_vote → resolution**.

**Acceptance criteria:**
- `setup`: распределение ролей, рассылка секретной информации.
- `main_round`: цикл вопросов/ответов, длится до таймера / лимита вопросов / триггера перехода в голосование.
- Досрочное голосование **не инициируется по желанию агента**. Оно запускается кодовым слоем при выполнении одного из условий (см. `vote_trigger_rules.json`):
  - один персонаж получил 2+ прямых обвинения
  - 2 агента подряд указали на одного и того же
  - N ходов без прогресса в расследовании
- `final_vote`: обязательно при достижении таймера или лимита вопросов, если партия ещё не завершилась иначе.
- `resolution`: определяется победитель, пишется финальная запись в лог.

### 3.7 F7. Эмуляция "проговаривания" реплик

Искусственная задержка перед показом реплики, пропорциональная её длине, для естественного темпа восприятия.

**Acceptance criteria:**
- После генерации реплики она не сразу отображается, а с задержкой `length * SPEECH_DELAY_MULTIPLIER + random(0.2s, 0.6s)`.
- Задержка применяется и к вопросам, и к ответам, и к вмешательствам.
- На период паузы в UI отображается индикатор "персонаж X печатает/говорит".

### 3.8 F8. Логирование партий

Каждая партия пишется в отдельный JSON-файл, никогда не перезаписывается.

**Acceptance criteria:**
- Имя файла: `games/{YYYY-MM-DD}_{HH-MM-SS}_{game_id}.json`.
- Содержит: метаданные партии, полный лог реплик с таймстемпами, внутреннее состояние (шкала уверенности шпиона по ходам, какие триггеры сработали), итог.
- Формат позволяет лёгкую выгрузку и дальнейшую обработку (монтаж, аналитика).

### 3.9 F9. Минимальный UI для отладки

Простой веб-интерфейс (после стабилизации ядра) для запуска, паузы и остановки партии.

**Acceptance criteria:**
- Кнопки: Start / Pause / Resume / Stop.
- Stop **немедленно** прерывает все inflight LLM-запросы (важно для контроля бюджета).
- Отображение живого чата партии с указанием кто → кому, таймера и статуса.
- Форматирование: имя говорящего, адресат (или "всем"), текст реплики.

### 3.10 F10. Конфигурация через env и JSON

Рантайм-параметры — через env, контент — через JSON-файлы.

**Acceptance criteria:**
- Env-переменные (см. раздел 10): ключи API, длительность, состав, модели, задержки.
- Контентные файлы: `characters/*.json`, `locations.json`, `trigger_rules.json`, `llm_config.json`.
- Изменение контентного файла не требует перезапуска билда — только перезапуска процесса (или hot-reload при желании, вне MVP).

## 4. Рекомендации по техническому стеку

### 4.1 Язык и рантайм

**Python 3.11+**. Причины: зрелая LLM-экосистема, простой async для оркестрации параллельных запросов к агентам, низкий барьер входа для расширения.

### 4.2 LLM-провайдеры и модели

**Стартовый провайдер:** OpenAI. **Основная модель:** GPT-4o (качество характера реплик). **Утилитарная модель:** GPT-4o-mini (микро-вызовы: проверка триггеров, решение "влезать / нет", шкала уверенности шпиона). Ожидаемая экономия на служебных вызовах — порядок 10×.

**Важно:** весь доступ к LLM изолируется через тонкий **адаптер-слой** (не LangChain, своя реализация ~50–100 строк). Адаптер поддерживает подмену провайдера: OpenAI / Anthropic / Google / любой другой совместимый. Выбор провайдера и модели — в `llm_config.json` и переопределяется через env.

**Структура `llm_config.json` (пример):**

```json
{
  "providers": {
    "openai": {
      "api_key_env": "OPENAI_API_KEY",
      "base_url": "https://api.openai.com/v1"
    },
    "anthropic": {
      "api_key_env": "ANTHROPIC_API_KEY",
      "base_url": "https://api.anthropic.com/v1"
    }
  },
  "roles": {
    "main": { "provider": "openai", "model": "gpt-4o" },
    "utility": { "provider": "openai", "model": "gpt-4o-mini" }
  },
  "per_character_overrides": {
    "boris_molot": { "provider": "openai", "model": "gpt-4o" }
  }
}
```

### 4.3 Веб-слой

После стабилизации ядра (фаза 2, см. раздел 9) — **FastAPI** для бэка + **минимальный HTML/JS** (один файл `index.html`, WebSocket для live-обновления чата). Фреймворк не нужен — прототипу достаточно vanilla JS.

### 4.4 Хранение

**JSON-файлы на диске.** Каждая партия — отдельный файл, не перезаписывается. Задел под БД оставляем через репозиторий-паттерн (абстракция над хранилищем), но реализуем только файловый репозиторий. Миграция на SQLite/Postgres позже — замена реализации, не интерфейса.

### 4.5 Структура проекта (предварительная)

```
spyfall-ai/
├── src/
│   ├── orchestrator/          # движок партии, фазы, ходы
│   ├── agents/                # сборка промптов, вызовы LLM
│   ├── triggers/              # проверка условий, приоритеты
│   ├── llm/                   # адаптер провайдеров
│   ├── storage/               # репозиторий логов партий
│   ├── web/                   # FastAPI + ws (фаза 2)
│   └── cli.py                 # точка входа для фазы 1
├── characters/                # JSON-профили персонажей
├── locations.json
├── trigger_rules.json
├── vote_trigger_rules.json
├── llm_config.json
├── games/                     # логи партий
├── .env.example
└── README.md
```

### 4.6 Обоснование отказов

- **Нет LangChain / LlamaIndex** — избыточны, добавляют когнитивную нагрузку и лишний слой.
- **Нет БД на MVP** — JSON достаточно, читаемо, дёшево, под задел оставлен репозиторий.
- **Нет React/Next.js** — один HTML-файл решает задачу полностью.
- **Нет очередей / Redis / воркеров** — партия проходит линейно в одном процессе.

## 5. Концептуальная модель данных

### 5.1 Character (персонаж)

**Файл:** `characters/{character_id}.json`

| Поле | Тип | Описание |
|---|---|---|
| `id` | string | Уникальный slug: `boris_molot`, `margo_honeytongue` и т.п. |
| `display_name` | string | Имя для показа в чате: "Борис", "Марго" |
| `archetype` | string | Короткий ярлык: "агрессор", "манипулятор" |
| `backstory` | string | Биография на 3–5 предложений, даёт мотивацию поведения |
| `voice_style` | string | Описание речевой манеры: лексика, длина фраз, тики |
| `must_directives` | string[] | Список MUST-правил игрового поведения (каждое — машинно проверяемое, см. 8.0) |
| `must_not_directives` | string[] | Список MUST NOT-правил (тоже проверяемых) |
| `detectable_markers` | Marker[] | 3–5 формальных признаков "характерной реплики" для метрик готовности (см. 8.0) |
| `personal_triggers` | Trigger[] | Персональные триггеры (см. 5.3) |
| `intervention_priority` | int (1–10) | Приоритет при конкуренции окна вмешательства |
| `intervention_threshold` | float (0–1) | Порог срабатывания (чем ниже — тем буйнее) |
| `llm_override` | object? | Опциональная подмена модели/провайдера |

### 5.2 Location (локация)

**Файл:** `locations.json` — массив объектов.

| Поле | Тип | Описание |
|---|---|---|
| `id` | string | slug: `hospital`, `airplane` |
| `display_name` | string | Название для показа |
| `description` | string | Короткое описание атмосферы (1–2 предложения) |
| `roles` | Role[] | 3–5 ролей |

**Role:**

| Поле | Тип | Описание |
|---|---|---|
| `id` | string | slug: `surgeon`, `patient` |
| `display_name` | string | Название роли |
| `description` | string | Короткое пояснение обязанностей |

### 5.3 Trigger (триггер)

Единая структура для глобальных и персональных триггеров.

| Поле | Тип | Описание |
|---|---|---|
| `condition_type` | enum | См. список ниже |
| `priority` | int (1–10) | Приоритет при конкуренции |
| `threshold` | float (0–1) | Чувствительность срабатывания |
| `reaction_type` | enum | См. список ниже |
| `params` | object? | Дополнительные параметры (например, `silent_turns: 2`) |

**`condition_type` (enum, MVP — только формально детектируемые условия):**

- `direct_accusation` — агент прямо обвинён (детектится по упоминанию имени + маркеры обвинения)
- `silent_for_n_turns` — агент молчит N ходов (параметр: `silent_turns`) — чисто счётчик ходов
- `dodged_direct_question` — агент получил прямой вопрос и не ответил по сути (детектится утилитарной моделью одним бинарным вызовом после ответа)
- `repeated_accusation_on_same_target` — один и тот же игрок обвинён 2+ раз (счётчик по игрокам)
- `contradiction_with_previous_answer` — текущий ответ противоречит ранее сказанному этим же агентом (детектится утилитарной моделью при наличии ранее сказанного на ту же тему). **Второй эшелон надёжности.** Склонен ложно срабатывать на смещение темы, расплывчатые формулировки, безобидный перефраз. Держать под наблюдением в фазе 3: если шумит — первый кандидат на временное отключение или повышение порога.

**Принцип отбора:** на MVP каждый `condition_type` должен быть либо чисто детерминистски детектируемым в коде (счётчики, имена), либо однозначным yes/no-запросом к утилитарной модели. Семантические триггеры типа "пафосная банальность", "моральное заявление", "уверенная ложь" — **вне MVP**: на бумаге красиво, в коде превращаются в муть. Добавляются после фазы 3, когда живой цикл работает и становится понятно, что реально нужно.

**`reaction_type` (enum, стартовый список):**

- `pressure_with_sharper_question` — давление уточняющим резким вопросом
- `mock_with_dry_sarcasm` — сарказм, высмеивание
- `moralize_and_accuse` — морализаторское обвинение
- `panic_and_derail` — нервный выпад, уводящий тему
- `point_out_inconsistency` — указать на противоречие
- `deflect_suspicion_to_another` — перевести подозрение на другого
- `short_dismissive_jab` — короткий обрубающий комментарий

**Расширение списков:** каждый новый `condition_type` и `reaction_type` добавляется кодом + документацией. Свободные строки не допускаются.

### 5.3.1 Marker (формальный признак характерной реплики)

| Поле | Тип | Описание |
|---|---|---|
| `id` | string | slug маркера: `short_rubbed_sentence`, `named_addressee_by_name`, `contains_analogy` |
| `method` | enum | `regex` / `counter` / `binary_llm` |
| `pattern` | string? | Регулярка или ключевые слова (для `regex`) |
| `rule` | string? | Описание счётчика (например, `sentences <= 2`) для `counter` |
| `prompt` | string? | Однозначный yes/no-промпт для `binary_llm` |
| `description` | string | Человекочитаемое пояснение, что значит срабатывание |

### 5.4 Game (партия)

**Файл:** `games/{timestamp}_{game_id}.json`

| Поле | Тип | Описание |
|---|---|---|
| `id` | string (uuid) | Уникальный ID партии |
| `started_at` | ISO timestamp | Время старта |
| `ended_at` | ISO timestamp? | Время окончания |
| `config` | object | Снимок конфигурации (модели, длительность, состав) |
| `location_id` | string | Выбранная локация |
| `players` | Player[] | Список участников |
| `spy_id` | string | ID шпиона |
| `turns` | Turn[] | Полный лог ходов |
| `spy_confidence_log` | ConfidenceEntry[] | История шкалы уверенности шпиона |
| `triggered_events` | TriggerEvent[] | Какие триггеры и когда сработали |
| `phase_transitions` | PhaseEntry[] | История фаз |
| `outcome` | object | Итог: кто победил, причина, голоса |

**Player:**

| Поле | Тип | Описание |

## 6. Принципы UI-дизайна (минимальный веб-UI для отладки)

### 6.1 Экран партии

Один экран, три зоны:

1. **Шапка** — название партии, локация (видна создателю, но не "зрителю" если мы решим разделять роли позже), таймер, фаза, кнопки управления.
2. **Чат** — хронологический поток реплик, каждая реплика содержит: имя + цвет персонажа, адресат, тип (вопрос/ответ/вмешательство), текст, таймстемп. Автоскролл при новой реплике.
3. **Сайдбар** (опционально) — список игроков с индикаторами: кто говорит сейчас, чья очередь, кто под подозрением (визуально: подсветка).

### 6.2 Принципы

- **Читаемость превыше красоты.** Монопространственный шрифт допустим.
- **Никакой магии.** Все состояния (pause / stop / wait) видны явно.
- **Контраст говорящих.** У каждого персонажа свой постоянный цвет.
- **Тип реплики явно различим.** Вопрос, ответ, вмешательство — разная визуальная подача (например, вопрос — обычный текст, вмешательство — курсив с отступом).

### 6.3 За пределами MVP

Аватары, анимации "говорящего", звуки, режим "зрителя" (где скрыта отладочная информация), театр-вью с позиционированием персонажей вокруг стола.

## 7. Соображения по безопасности

### 7.1 API-ключи

- Ключи только через `.env`, файл в `.gitignore`.
- `.env.example` коммитится с пустыми значениями.
- Никаких ключей в логах партий, коде или UI.

### 7.2 Ограничение расходов

- Жёсткий kill-switch: Stop в UI должен отменять inflight-запросы (через `asyncio.CancelledError` или `AbortController`).
- В env — `MAX_PARTY_COST_USD` (опционально), после превышения партия останавливается автоматически.
- Счётчик токенов и стоимости в логе партии (обновляется после каждого вызова).

### 7.3 Локальный запуск

MVP не выставляется в публичный интернет. Веб-UI слушает только `127.0.0.1`. Если понадобится удалённый доступ — добавить авторизацию и HTTPS (вне MVP).

### 7.4 Контент

Персонажи получают директивы, которые могут включать обвинения, сарказм, моральное давление — это **внутриигровое поведение**, не реальные атаки. В промптах каждого персонажа явно фиксировать: MUST NOT использовать реальные оскорбления по защищённым признакам, только игровое давление в рамках Spyfall. Это не "цензура характера", а отделение игрового конфликта от реальной токсичности.

## 8. Спецификация персонажей (стартовая труппа)

### 8.0 Сквозной принцип: проверяемость директив и character marker

Два правила, которые обязательны для всех профилей персонажей на MVP. Они определяют форму MUST/MUST NOT и связаны с метриками готовности (раздел 14).

**Правило 1. Каждый MUST / MUST NOT обязан быть машинно проверяемым.**

Проверяемость значит одно из двух:
- регуляркой или детерминистской эвристикой в коде (например, "не более 2 предложений" — считаем точки/восклицания/вопросительные; "обратился по имени" — поиск имени другого игрока в реплике);
- одним бинарным yes/no-вызовом к утилитарной модели с коротким однозначным промптом ("содержит ли реплика аналогию или сравнение? да/нет").

Если директиву нельзя сформулировать так, чтобы её реализация определялась автоматически — это **не MUST, а описание стиля**, и она переносится в поле `voice_style`, где живёт только для промпта агента, без претензий на проверку.

Пример преобразования. Исходное "В качестве шпиона — применять тот же приём против честных игроков" непроверяемо. В MUST этого писать нельзя. Переформулировка: "В каждом своём ответе обратиться к собеседнику по имени" (проверяемо) + "используй моральное давление, особенно когда чувствуешь ложь" (уходит в `voice_style`).

Эта требовательность к форме MUST — цена за автоматизированную проверку характерности партии в разделе 14.

**Правило 2. Character marker — отсутствие маркера как метрика.**

Character marker реплики персонажа — это **любой формально детектируемый признак** того, что реплика соответствует хотя бы одному из его MUST-паттернов или его `voice_style` (в части, которую тоже удалось формализовать — например, характерная лексика, длина фраз, маркерные слова).

Для каждого персонажа в его профиле определяется `detectable_markers` — список из 3–5 таких признаков с указанием метода проверки (regex / счётчик / бинарный LLM-вызов). Реплика считается **немаркированной**, если ни один маркер не сработал. Это не вкусовая оценка "нейтрально ли звучит", а объективное отсутствие попадания в список.

Именно это определение используется в критериях готовности (раздел 14): "не было 2+ подряд немаркированных реплик у одного персонажа".

Ниже — архетипическая скелетная спецификация для каждого из 8 персонажей. Финальные списки `detectable_markers` и точные формулировки MUST допиливаются при реализации фазы 1, следуя двум правилам выше. Директивы, которые в текущей редакции нарушают правило проверяемости, помечены как кандидаты на перенос в `voice_style` и будут переформулированы на этапе имплементации.

## 9. Этапы разработки / milestones

### Фаза 0. Микро-партия end-to-end (1–1.5 недели)

**Цель фазы:** не "красивый технический каркас, который мало что доказывает", а минимальная полная партия Spyfall, проходящая от начала до конца. Упрощения — только там, где без них на старте никак; все упрощения компенсируются в фазе 1.

**Что входит:**
- Структура проекта, `llm_config.json`, адаптер провайдера (1 провайдер — OpenAI).
- **3 персонажа** с упрощёнными, но реальными профилями (не заглушки-болванчики, а настоящие Борис / Зоя / Ким из финальной труппы с урезанными MUST/MUST NOT — по 2–3 директивы). Один из троих назначается шпионом.
- **2 локации** с 3 ролями каждая.
- Реальный цикл Spyfall: setup → round-robin вопрос/ответ → final_vote → resolution. Голосование — простейшее, по таймеру или лимиту вопросов.
- CLI-запуск, вывод партии в консоль, логирование в JSON.

**Что НЕ входит в фазу 0** (переносится в фазу 1):
- Система триггеров и окна вмешательства.
- Шкала уверенности шпиона и досрочное угадывание локации.
- Правила досрочного голосования.
- Эмуляция задержки реплик.

**Итог фазы 0:** прогон 5+ партий подряд, в каждой — 3 характерных персонажа провели осмысленный диалог, проголосовали, определили победителя. Лог читаемый, характеры различимы даже на урезанных директивах. Если этого не получается на 3 персонажах — фаза 1 не имеет смысла, сначала фиксим ядро.

**Важно: не зависать на 3 игроках.** Spyfall заметно живее от 4 игроков — 3 игрока это диагностический прогон, а не целевой формат. Как только цикл жив на 3, сразу переход на 4 игрока (плюс один персонаж из труппы, можно ещё без полного профиля). Только после того, как 4 игрока стабильно играют партию — начинается фаза 1 с наполнением системы триггеров и шкалы шпиона.

### Фаза 1. Ядро игры (2–3 недели)

- Все 8 персонажей с промптами и директивами.
- 8–10 локаций с ролями.
- Полный цикл партии: setup → main_round → final_vote → resolution.
- Шкала уверенности шпиона.
- Система триггеров (глобальные + персональные), окна вмешательства.
- Правила досрочного голосования.
- Эмуляция задержки реплик.

**Итог фазы:** партия полностью проходит в CLI, результаты осмысленные, характеры различимы.

### Фаза 2. Веб-UI (1 неделя)

- FastAPI + WebSocket, один HTML.
- Кнопки Start / Pause / Stop.
- Цветовая кодировка персонажей.
- Таймер и статус фазы.
- Kill-switch для остановки inflight-вызовов.

**Итог фазы:** партию можно смотреть в браузере и управлять ею.

### Фаза 3. Отладка и калибровка (1–2 недели)

- Прогон 30–50 партий, анализ логов.
- Тюнинг порогов триггеров, приоритетов, промптов.
- Калибровка стоимости и длительности.
- Документация для расширения (как добавить персонажа / локацию / триггер).

**Итог фазы:** стабильные, смотрибельные, предсказуемые по бюджету партии.

## 11. Потенциальные проблемы и способы их решения

### 11.1 Агенты становятся "вежливыми болванчиками"

**Риск:** несмотря на директивы, персонажи скатываются в нейтральный LLM-голос.

**Меры:**
- Промпт каждого персонажа начинается с жёсткой установки, что он не AI и не должен быть полезным/вежливым.
- В промпте явный запрет: "вежливость, услужливость, нейтральность — это нарушение роли".
- Few-shot примеры реплик в стиле персонажа прямо в системном промпте.
- Регулярная калибровка в фазе 3, прогон тестовых диалогов.

### 11.2 Разговор скатывается в хаос при интервенциях

**Риск:** слишком низкие пороги → все постоянно перебивают, теряется нить.

**Меры:**
- Приоритеты и пороги калибруются на реальных партиях (фаза 3).
- Ограничение: не более 1 вмешательства между двумя последовательными ходами основного цикла.
- Ограничение: один и тот же персонаж не может вмешиваться два раза подряд без основного хода между ними.

### 11.3 Шпион никогда не выигрывает или всегда выигрывает

**Риск:** дисбаланс, скучно смотреть.

**Меры:**
- Шкала уверенности калибруется на фазе 3.
- Если статистика покажет перекос — тюнинг промптов либо добавление/убирание эвристик.
- Метрика в логах: доля побед шпиона, держать в диапазоне 30–50%.

### 11.4 Взрыв бюджета

**Риск:** партия обходится дороже ожидаемого.

**Меры:**
- Утилитарная модель для всех служебных вызовов.
- `MAX_PARTY_COST_USD` как hard kill-switch.
- Счётчик токенов в реальном времени в логе.
- Конспект истории партии после каждых N ходов (compression), чтобы не тащить весь лог в контекст.

### 11.5 Длинный контекст удорожает каждый следующий ход

**Риск:** 80-я реплика стоит в 10 раз дороже первой.

**Меры:**
- Формирование контекста для агента: последние K полных ходов + конспект более ранних.
- Периодическое сжатие истории через утилитарную модель.

### 11.6 Персонажи путаются в ролях (забывают, что они шпион / не шпион)

**Риск:** агент-шпион случайно упоминает локацию, агент знает свою роль, но игнорирует её.

**Меры:**
- Секретная информация отправляется в системном промпте, обновляется на каждом вызове.
- Явное напоминание: "ты шпион, ты НЕ знаешь локацию" / "ты работаешь в больнице хирургом".
- Автоматическая детекция прямого упоминания локации шпионом (post-check) — если случилось, считается провалом шпиона, партия завершается.

### 11.7 Генерация скучных вопросов "в лоб"

**Риск:** агенты задают "это больница?" — партия сломана.

**Меры:**
- В базовом промпте каждого агента — явное правило про непрямые вопросы, с несколькими few-shot примерами хороших и плохих вопросов.
- Утилитарная пост-проверка вопросов: если модель детектит прямое упоминание локации в вопросе мирного — реролл.

## 12. Возможности будущего расширения

### 12.1 Контент и персонажи

- Увеличение труппы до 20–30 персонажей.
- Генерация персонажей под партию (гибрид "ядро + генерация").
- Расширение каталога локаций до 30+.
- Поддержка кастомных локаций от сообщества.

### 12.2 Продукт

- Live-стриминг партий с публичным доступом.
- Библиотека записанных партий с поиском по персонажам / локациям / типу финала.
- Клипирование: автоматическое выделение "горячих моментов" партии.
- Участие живого игрока среди AI.
- Агент-нарратор / ведущий / комментатор.

### 12.3 Визуал и презентация

- Аватары персонажей (статические → анимированные).
- Театральный вид: персонажи вокруг стола, спикер подсвечивается.
- TTS-озвучка реплик с индивидуальными голосами.
- Видео-рендер партий для загрузки.

### 12.4 Техника

- Миграция с JSON на SQLite / Postgres для аналитики.
- Мультимодельный mix: разные провайдеры под разных персонажей для разнообразия "голоса".
- Fine-tuning собственных моделей на успешных партиях.
- Автоматическая калибровка порогов триггеров на основе метрик партий.

### 12.5 Монетизация (вне MVP)

- Подписка на приватные партии с кастомным составом.
- Маркетплейс персонажей / локаций.
- Турниры AI-агентов с голосованием зрителей.

## 13. Открытые вопросы для ревью

Пункты, которые намеренно оставлены на решение в процессе разработки:

1. **Точные пороги и приоритеты триггеров** — калибруются в фазе 3 на реальных партиях.
2. **Точный лимит вопросов до голосования** — зависит от темпа партий, фиксируется эмпирически.
3. **Формат few-shot примеров в промптах** — будет разработан при реализации фазы 1.
4. **Конкретные локации и роли** — стартовый каталог пишется одновременно с персонажами, зависит от культурного контекста (подбираем те, где архетипы ярко взаимодействуют).
5. **Протокол WebSocket-сообщений** — фиксируется на входе в фазу 2.

## 14. Критерии готовности MVP

MVP считается готовым, когда:

- [ ] Партия из 4 персонажей проходит полный цикл без сбоев.
- [ ] Партия из 6 персонажей проходит полный цикл без сбоев.
- [ ] Все 8 персонажей имеют различимый "голос" (проверка: человек, слушая лог, может назвать говорящего по стилю).
- [ ] **Формальная проверка характерности партии** (обе метрики должны выполняться на 80%+ партий из тестовой серии):
  - Каждый персонаж партии минимум **2 раза** проявил свой MUST-паттерн (детектится автоматически по логу — возможно благодаря сквозному правилу "MUST обязан быть машинно проверяемым", см. раздел 8.0).
  - У одного и того же персонажа **не было 2+ подряд немаркированных реплик**, где "немаркированная" означает "ни один из `detectable_markers` персонажа не сработал" (см. раздел 8.0), а не субъективное "нейтрально звучит".
- [ ] Шпион выигрывает в 30–50% партий на длинной серии (20+ прогонов).
- [ ] Партия длительностью 20 минут стоит не более $2.
- [ ] Веб-UI корректно отображает партию, Stop немедленно прерывает.
- [ ] Лог партии полностью восстанавливает её ход, включая внутренние состояния.
- [ ] Добавление нового персонажа занимает не более 1 часа (только JSON + тесты).
