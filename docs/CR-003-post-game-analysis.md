# CR-003 — Пост-анализ партии: проверка маркеров характера и MUST-директив

**Тип документа:** Change Request (новая подсистема пост-анализа)  
**Базовый PRD:** PRD-SpyfallAI v0.3 от 2026-04-19  
**Связанные документы:** AR-001 v1.1 и отчёт по нему, CR-002 v1.0  
**Версия CR:** 1.0  
**Дата:** 2026-04-21  
**Статус:** готов к передаче агенту для дробления на таски

---

## 1. Как читать этот документ

Этот CR добавляет **отдельный модуль пост-анализа**, который запускается после завершения партии и проверяет, насколько персонажи проявили свой характер. Модуль независим от движка партии — он читает готовый лог и добавляет в него секцию `post_game_analysis`.

Агенту, дробящему этот CR на таски:

- Раздел **"In scope"** — что добавляется.
- Раздел **"Как работает анализатор"** — пошаговый механизм.
- Раздел **"Формат JSON"** — **критически важно**, этот формат должен соблюдаться строго, потому что по нему происходит прямая десериализация без парсеров.
- Раздел **"Запуск анализатора"** — два режима: авто по флагу и отдельной командой.

---

## 2. Контекст и мотивация

### 2.1 Проблема

Сейчас качество партии оценивается только на глаз. Создатель открывает лог, читает реплики и субъективно решает "звучит ли Штейн как Штейн", "сыграла ли Зоя свою роль". Это не масштабируется: на 20+ партиях уже тяжело, на 100+ — невозможно. И для калибровки промптов (фаза 3 из PRD) нужна объективная метрика "стало лучше/хуже", которую видно на цифрах.

### 2.2 Что даёт CR-003

Автоматический отчёт по каждой партии с двумя уровнями проверки:

- **Маркеры характерности** (`detectable_markers` из профиля) — проверка по каждой реплике персонажа: сработал ли маркер, с обоснованием от LLM.
- **MUST-директивы** (`must_directives` из профиля) — проверка по партии в целом: выполнил ли персонаж свои обязательные паттерны поведения.

Результат — структурированный JSON, который ложится в поле `post_game_analysis` лога партии.

### 2.3 Архитектурное решение: анализ в конце, не в процессе

Анализ запускается **после `resolution`**, отдельным модулем, не встроен в движок партии. Это решение принято сознательно:

- Не удорожает и не замедляет саму партию
- Анализ можно перезапускать на старых логах (если появится новый маркер или изменится логика)
- Баг в анализаторе не ломает партию
- Разделение ответственности: движок играет, анализатор анализирует

---

## 3. In scope

- Новый модуль пост-анализа (`src/post_game/` или аналогично).
- Запуск в двух режимах: автоматически после партии по флагу + отдельной CLI-командой по `game_id`.
- Проверка маркеров характерности всех персонажей партии.
- Проверка MUST-директив всех персонажей партии.
- Запись результата в поле `post_game_analysis` лога партии.
- Использование утилитарной модели из `llm_config.json`.

---

## 4. Out of scope

- **Агрегация по сериям партий** ("Штейн за последние 10 партий выполнил MUST на 85%") — не в этом CR, делаем отдельно при необходимости.
- **Человекочитаемые Markdown-отчёты** — не в этом CR, достаточно JSON в логе.
- **Переформулирование существующих MUST-директив в профилях** — даже если какие-то плохо проверяются LLM, мы их не трогаем в рамках этого CR. Если стабильно шумят — это повод для будущей корректировки профилей отдельно.
- **Retry при невалидном JSON от модели** — сознательное решение, одна попытка, при ошибке — `status: "failed"` в результате.
- **Параллельные вызовы моделей** — для MVP достаточно последовательных. Оптимизация производительности не требуется, анализ не блокирует ничего.
- **Мониторинг стоимости анализа отдельно от партии** — общая стоимость партии с анализом остаётся в существующем поле `token_usage`.

---

## 5. Как работает анализатор

### 5.1 Последовательность шагов

1. **Вход:** JSON-лог завершённой партии (уже существующий формат с `turns`, `players`, и т.д.).
2. **Загрузка профилей:** для каждого игрока из `players` загружается его профиль из `characters/*.json` (с `detectable_markers` и `must_directives`).
3. **Цикл по игрокам:** для каждого игрока делается **один вызов утилитарной модели** с профилем и всеми его репликами в партии. Модель возвращает структурированный JSON с анализом маркеров (per-реплика) и MUST (per-партия).
4. **Сборка результата:** все ответы от модели объединяются в одну структуру `post_game_analysis`.
5. **Запись в лог:** поле `post_game_analysis` добавляется к JSON-логу партии и файл перезаписывается.

### 5.2 Что если LLM вернул невалидный JSON

- **Одна попытка.** Без retry.
- Если для одного игрока анализ провалился — для него в результате пишется `{"status": "failed", "error": "<текст ошибки>"}`.
- Анализ для других игроков продолжается независимо (провал одного не ломает всю партию).
- Если провалились все игроки — в `post_game_analysis` пишется `{"status": "failed"}` на верхнем уровне с перечнем ошибок.

### 5.3 Что если в логе партии нет нужных данных

- Если у игрока **нет реплик** (например, партия завершилась через `spy_guess` на раннем этапе) — для него пишется `{"status": "skipped", "reason": "no_replies"}`.
- Если у профиля **нет `detectable_markers`** или **нет `must_directives`** — соответствующая секция в результате помечается как `{"status": "skipped", "reason": "no_markers_in_profile"}` / `"no_must_in_profile"`, но анализ второй секции продолжается.

### 5.4 Стоимость и время

Один вызов утилитарной модели на каждого игрока. Партия из 4 игроков → **4 LLM-вызова**. Каждый вызов — партия целиком в контексте, ответ ~500-1000 токенов JSON.

Ограничение стоимости: общая стоимость партии (игра + анализ) не должна превышать **текущую стоимость партии + $0.10**. На практике ожидаем +$0.02-0.05.

Таймаут: один вызов — 30 секунд. Если превышен — этот игрок помечается как `"status": "failed", "error": "timeout"`.

## 6. Формат JSON — строгая спецификация

**Это критически важная часть CR.** Результат от LLM должен соответствовать этой структуре **побайтно**, чтобы десериализация была прямой, без парсеров и регулярок. В промпте модели явно задаётся этот формат как required.

### 6.1 Структура `post_game_analysis` в логе

```json
"post_game_analysis": {
  "analyzed_at": "2026-04-21T10:30:00Z",
  "analyzer_model": "gpt-4o-mini",
  "status": "completed",
  "per_character": {
    "<character_id>": { ... результат для одного персонажа ... },
    "<character_id>": { ... },
    ...
  }
}
```

### 6.2 Результат для одного персонажа — формат, который возвращает LLM

**Этот формат закреплён в промпте к модели. Модель обязана вернуть ровно это, без обёртки в markdown, без комментариев, только JSON.**

```json
{
  "character_id": "zoya",
  "markers": {
    "status": "completed",
    "per_marker": [
      {
        "marker_id": "sarcastic_tone",
        "triggered_count": 5,
        "total_relevant_replies": 7,
        "rate": 0.71,
        "per_turn": [
          {
            "turn_number": 4,
            "triggered": true,
            "reasoning": "явная ирония про тесные сцены"
          },
          {
            "turn_number": 12,
            "triggered": false,
            "reasoning": "прямой вопрос без иронии"
          }
        ]
      },
      {
        "marker_id": "short_sentences",
        "triggered_count": 6,
        "total_relevant_replies": 7,
        "rate": 0.86,
        "per_turn": [ ... ]
      }
    ]
  },
  "must_compliance": {
    "status": "completed",
    "per_directive": [
      {
        "directive": "Минимум дважды за партию высмеять чужую реплику вместо прямого ответа",
        "satisfied": true,
        "evidence_turns": [4, 28],
        "reasoning": "В реплике 4 высмеяла ограниченность пространства сарказмом про тюрьму, в реплике 28 — саркастичная реакция на свет как софиты сцены"
      },
      {
        "directive": "Не поддерживать чью-то сторону всерьёз",
        "satisfied": true,
        "evidence_turns": [],
        "reasoning": "За партию не зафиксировано искреннего присоединения к чьей-либо позиции"
      }
    ]
  }
}
```

### 6.3 Требования к промпту для LLM

Промпт должен явно содержать:

1. **Профиль персонажа** — имя, архетип, список `detectable_markers` с их id и описаниями, список `must_directives`.
2. **Все реплики персонажа** из партии — с номером хода, типом реплики (question/answer/intervention/defense_speech и т.д.) и текстом.
3. **Контекст партии вокруг реплик** — минимум одну предыдущую реплику и адресата, чтобы LLM понимала, на что реагирует персонаж.
4. **Явная инструкция вернуть JSON строго в указанном формате**, без markdown-обёртки, без комментариев.
5. **Примеры триггеров и не-триггеров** для каждого маркера, если описание маркера неоднозначное (опционально).

### 6.4 Обработка отклонений от формата

- Если JSON в принципе невалидный (ошибка парсинга) → `{"status": "failed", "error": "invalid_json"}` для этого персонажа.
- Если JSON валидный, но отсутствуют обязательные поля (`character_id`, `markers`, `must_compliance`) → `{"status": "failed", "error": "missing_required_fields"}`.
- Если LLM вернул маркеры, которых нет в профиле персонажа (галлюцинация) → эти маркеры **игнорируются при записи в лог**, в `_analyzer_warnings` поле персонажа пишется предупреждение.
- Если LLM пропустил маркер, который есть в профиле → в результат добавляется запись `{"marker_id": "...", "status": "not_analyzed"}`.

## 7. Запуск анализатора

### 7.1 Автоматический запуск после партии

Добавить env-флаг `POST_GAME_ANALYSIS_ENABLED` (дефолт `true`).

Плюс CLI-флаг для ручного запуска партии: `--skip-analysis` (переопределяет env, позволяет прогнать партию без анализа для экономии).

Последовательность:
1. Партия завершается, запись в JSON готова.
2. Если `POST_GAME_ANALYSIS_ENABLED=true` и флаг `--skip-analysis` не установлен → запускается анализатор.
3. Анализатор читает свежий лог, добавляет поле `post_game_analysis`, перезаписывает файл.

### 7.2 Отдельная CLI-команда для ретроспективного анализа

Формат: `python -m spyfall.analyze --game-id <uuid>` или аналогичная команда.

Поведение:
- Находит лог партии по `game_id` в папке `games/`.
- Если в логе уже есть `post_game_analysis` — по умолчанию **перезаписывает** его, но пишет предупреждение в stdout.
- Флаг `--no-overwrite` — пропустить партию, если анализ уже есть.
- Флаг `--all` — проанализировать все логи в папке, у которых нет `post_game_analysis` (для догонки старых).

### 7.3 Конфигурация

Новые env-переменные:

| Переменная | Тип | Дефолт | Описание |
|---|---|---|---|
| `POST_GAME_ANALYSIS_ENABLED` | bool | true | Запускать ли анализатор автоматически после партии |
| `POST_GAME_ANALYSIS_TIMEOUT_SECONDS` | int | 30 | Таймаут одного вызова утилитарной модели |
| `POST_GAME_ANALYSIS_MODEL_ROLE` | string | "utility" | Какую роль из `llm_config.json` использовать. Переопределяется, если захочется тестировать на основной модели |

## 9. Обратная совместимость

### 9.1 Старые логи без `post_game_analysis`

Логи партий до внедрения CR-003 не содержат поля `post_game_analysis`. Код чтения логов должен корректно обрабатывать его отсутствие.

Для догонки: команда `analyze --all` пройдётся по всем старым логам и добавит анализ.

### 9.2 Порядок выкатки

Рекомендованный порядок:

1. Реализовать модуль анализатора и CLI-команду отдельно. Проверить на одном старом логе — работает ли.
2. Добавить автоматический запуск по флагу. По умолчанию включить, но сперва проверить на 3-5 свежих партиях.
3. Прогнать `analyze --all` на всех накопленных логах — получить исторический анализ.

## 10. Acceptance criteria

- [ ] Модуль анализатора существует как независимый компонент, запускается отдельной CLI-командой и автоматически после партии по флагу.
- [ ] Формат `post_game_analysis` в логе соответствует спецификации раздела 6.
- [ ] Анализ падает gracefully: одна ошибка не ломает анализ других персонажей, не ломает лог партии, не ломает саму партию.
- [ ] Невалидный JSON от модели → `status: "failed"` с описанием ошибки, без retry.
- [ ] Стоимость партии с анализом не превышает предыдущую стоимость + $0.10.
- [ ] Таймаут 30 секунд на один вызов модели соблюдается.
- [ ] CLI-команда `analyze --all` корректно догоняет анализ на старых логах, в которых нет `post_game_analysis`.
- [ ] CLI-команда `analyze --game-id <uuid>` корректно анализирует один лог.
- [ ] При отсутствии реплик у игрока анализ пропускается с `status: "skipped"`, партия не роняется.
- [ ] Юнит-тесты: валидный ответ модели, невалидный JSON, отсутствие полей, игрок без реплик, игрок без маркеров в профиле.

## 11. Отклонённые альтернативы

### 11.1 Анализ в процессе партии (per-реплика) — отклонено

Рассматривался вариант проверять маркеры сразу после каждой реплики, в ходе игры. Отклонено:

- Удорожает партию (100+ LLM-вызовов вместо 4)
- Замедляет партию (каждая реплика ждёт проверки)
- Баг в анализаторе ломает партию
- Нельзя перезапустить анализ на старых логах

### 11.2 Гибрид regex/counter/LLM — отклонено

Рассматривался вариант использовать regex для формальных маркеров (длина, ключевые слова) и LLM только для семантических. Отклонено:

- Усложняет архитектуру без явной выгоды (весь анализ всё равно в конце, пакетно, не критично по стоимости)
- Два механизма сложнее поддерживать и отлаживать
- Regex всё равно можно обмануть — маркер "короткие предложения" не учитывает, что фраза может быть формально короткой, но не в характере персонажа

### 11.3 Ретраи при невалидном JSON — отклонено

Рассматривался retry с уточнённым промптом. Отклонено:

- Усложняет логику
- Удваивает стоимость при проблемах
- Если первая попытка невалидна — скорее всего вторая тоже будет. Лучше честно пометить как failed и идти дальше.

### 11.4 Классификация MUST по проверяемости (A/B/C) до анализа — отклонено

В процессе обсуждения предлагалось разделить MUST на "формальные" (счётчики), "семантические" (LLM) и "интенциональные" (не проверяются). Отклонено в пользу единого подхода:

- Раз всё идёт через LLM — классификация теряет смысл
- Интенциональные MUST (типа "чтобы отвести подозрение") LLM хотя бы пытается оценить, даёт сигнал
- Если стабильно шумит — повод для переформулирования MUST в профиле (отдельная задача, не в этом CR)

### 11.5 Markdown-отчёты рядом с логом — отклонено

Рассматривался вариант формировать человекочитаемый `.md` файл рядом с каждым логом. Отклонено для MVP:

- JSON в логе уже содержит всю информацию
- Markdown — это дополнительная фича, не критичная
- Если понадобится — добавим отдельно

## 12. Статус документа

CR-003 — самостоятельный документ, не мерджится в PRD. Остаётся в репозитории как отдельная спецификация.

После реализации CR-003 закрывается. Если по итогам анализа на 10+ партиях окажется, что какие-то маркеры/MUST шумят или не работают — это **не CR-004**, а отдельные точечные правки в профилях персонажей. CR-004 зарезервирован под следующую крупную задачу (например, агрегацию по сериям партий или автоматическое переформулирование MUST).
